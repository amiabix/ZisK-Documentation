Compile the guest to RISC-V, run it in the emulator, verify constraints, and generate a proof.

## Build the guest

From the `guest/` directory:

```bash
cd guest && cargo-zisk build --release && cd ..
```

This cross-compiles the guest into a RISC-V ELF binary at `target/riscv64ima-zisk-zkvm-elf/release/fibonacci-guest`.

## Prepare input

The guest reads `n` as a `u32` via `io::read()`. Create a binary input file with `n = 10`:

```bash
printf '\x0a\x00\x00\x00' > input.bin
```

That's the number 10 encoded as a little-endian `u32` (4 bytes).

## Run in the emulator

```bash
cargo-zisk execute \
  -e target/riscv64ima-zisk-zkvm-elf/release/fibonacci-guest \
  -i input.bin -l
```

`-l` selects the emulator backend (works on all platforms). On Linux, omit it for the faster ASM backend.

## Verify constraints

Check that the execution trace satisfies all PIL constraints before spending time on a full proof:

```bash
cargo-zisk verify-constraints \
  -e target/riscv64ima-zisk-zkvm-elf/release/fibonacci-guest \
  -i input.bin -l
```

Expected output:

```
[INFO ] CstrVrfy: All global constraints were successfully verified
[INFO ] CstrVrfy: All constraints were verified
```

If a constraint fails, there's a bug in the trace. Better to catch it here than after a full prove.

## Generate a proof

```bash
cargo-zisk prove \
  -e target/riscv64ima-zisk-zkvm-elf/release/fibonacci-guest \
  -i input.bin -o /tmp/fib-proof -l -a
```

`-a` enables aggregation, which produces the final VADCOP proof. Without it, only inner proofs are generated.

Expected output:

```
[INFO ] ✓ Vadcop Final proof was verified
```

See the [CLI Reference](/cli-reference) for all flags.
