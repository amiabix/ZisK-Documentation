---
title: Precompiles
description: Native VM instructions for SHA-256, Keccak, elliptic curve arithmetic, and other expensive operations.
---

Without precompiles, a SHA-256 compression expands into thousands of RISC-V instructions, each a row in the execution trace. Precompiles execute these operations as single native VM instructions, producing the same result with a dramatically smaller trace.

## How precompiles work

On a precompile call, ZisK reads inputs from guest memory, executes the operation natively, writes the result back, and records a single trace entry. The guest program sees the same result it would get from a software implementation, but the execution trace is orders of magnitude smaller.

Precompiles are triggered through special CSR instructions that ZisK's syscall layer emits. To use them, your guest program links against ZisK's patched forks of standard crypto crates, which replace internal implementations with these syscall instructions. The [Using precompiles](#using-precompiles) section below covers how to set this up.

## Using precompiles

To activate precompiles, use ZisK's patched forks of standard crates via `[patch.crates-io]` in your guest's `Cargo.toml` (or workspace root). The patched forks replace internal implementations with ZisK syscalls that trigger native precompile execution.

The patch must point to a **tag or branch that matches the version** your program depends on. Use `cargo tree` to verify that the patched fork is being resolved correctly.

### Example: SHA-256 with precompile

First, add the standard crate version under `[dependencies]`, then patch it to the ZisK fork under `[patch.crates-io]` with a matching version tag:

```toml
[dependencies]
sha2 = "0.10.9"
ziskos = { git = "https://github.com/0xPolygonHermez/zisk.git" }

[patch.crates-io]
sha2 = { git = "https://github.com/0xPolygonHermez/zisk-patch-hashes.git", tag = "patch-sha2-0.10.9-zisk-0.15.0" }
```

Verify the patch is applied:

```bash
cargo tree -i sha2
```

The output should show the `sha2` dependency pointing to the ZisK fork, not crates.io.

Your guest code uses the standard API:

```rust
use sha2::{Sha256, Digest};

fn main() {
    let data = b"hello, world";
    let hash = Sha256::digest(data);
    println!("hash: {:x}", hash);
}
```

The patched `sha2` fork calls `syscall_sha256_f` under the hood, which emits a CSR instruction that ZisK's transpiler replaces with the native precompile. The result is identical, but the execution trace is much smaller.

::callout{icon="i-heroicons-exclamation-triangle"}
Without the `[patch.crates-io]` entry, the standard `sha2` crate compiles and produces correct results but runs as software RISC-V instructions, **not** through the precompile. The trace will be much larger.
::

### Example: multiple patches

A real-world example patching multiple crates (from [zisk-eth-client](https://github.com/0xPolygonHermez/zisk-eth-client)):

```toml
[dependencies]
sha2 = "0.10.9"
tiny-keccak = "2.0.2"
k256 = "0.13.4"
blst = "=0.3.15"
ziskos = { git = "https://github.com/0xPolygonHermez/zisk.git" }

[patch.crates-io]
sha2 = { git = "https://github.com/0xPolygonHermez/zisk-patch-hashes.git", tag = "patch-sha2-0.10.9-zisk-0.15.0" }
tiny-keccak = { git = "https://github.com/0xPolygonHermez/zisk-patch-tiny-keccak.git", tag = "patch-2.0.2-zisk-0.15.0" }
k256 = { git = "https://github.com/0xPolygonHermez/zisk-patch-elliptic-curves.git", tag = "patch-k256-0.13.4-zisk-0.15.0" }
blst = { git = "https://github.com/0xPolygonHermez/zisk-patch-blst.git", tag = "patch-0.3.15-zisk-0.15.0" }
ruint = { git = "https://github.com/0xPolygonHermez/zisk-patch-ruint.git", branch = "zisk-patch/v1.17.2" }
substrate-bn = { git = "https://github.com/0xPolygonHermez/zisk-patch-bn.git", branch = "zisk-patch-bn/v0.6.0" }
aurora-engine-modexp = { git = "https://github.com/0xPolygonHermez/zisk-patch-modexp.git", branch = "zisk-patch/v1.2.0" }
```

The version in `[dependencies]` must match the version the patch was built for. Use `cargo tree` to verify each patched crate resolves to the ZisK fork.

## DMA macros

ZisK provides macros for DMA precompiles when manipulating large memory blocks:

```rust
use ziskos::{dma_memcpy, dma_memset, dma_memmove, dma_memcmp};

fn main() {
    let src = [1u8; 1024];
    let mut dst = [0u8; 1024];

    // Copy 1024 bytes from src to dst
    dma_memcpy(dst.as_mut_ptr(), src.as_ptr(), 1024);

    // Set 512 bytes to zero
    dma_memset(dst.as_mut_ptr(), 0, 512);
}
```

Behave like their C standard library counterparts but execute through DMA precompiles, compressing operations into a small number of trace rows.

## Impact on proving

Precompiles reduce trace rows, directly reducing proving time. The impact is most significant for heavy cryptography. ECDSA verification, for example, involves many field operations that each collapse to a single native operation.

If proving is slow, check whether your most expensive operations have precompile coverage. Make sure you're using the patched forks via `[patch.crates-io]` to activate precompile routing.
