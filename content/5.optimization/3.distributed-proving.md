A single machine can prove many programs comfortably, but some workloads -- programs with millions of execution steps, or production systems that need to prove continuously -- benefit from distributing the work. ZisK supports two levels of distribution:

- **MPI** splits proving across multiple CPU processes on a single machine.
- **Coordinator-worker** distributes proving across multiple machines over a network.

These two approaches can be combined: each worker machine in a cluster can use MPI internally to parallelize across its own cores and GPUs.

## MPI proving

MPI (Message Passing Interface) is the simplest way to parallelize proving. You run the same prover binary with multiple processes, and ZisK automatically divides the work among them.

### Running with MPI

From the command line:

```bash
mpirun -np 4 cargo-zisk prove -e path/to/elf -i input.bin -o /tmp/proof
```

The `-np 4` flag launches four processes. Each process handles a portion of the execution trace, generates partial proofs, and the results are combined automatically.

### MPI from the SDK

When using the SDK, build your prover normally:

```rust
use anyhow::Result;
use zisk_sdk::ProverClient;

fn main() -> Result<()> {
    let client = ProverClient::builder()
        .emu()
        .build()
        .unwrap();

    let (pk, vk) = client.setup(&ELF)?;
    let result = client.prove(&pk, stdin).run()?;
    println!("Proof generated successfully");

    Ok(())
}
```

Then run your compiled binary with `mpirun`:

```bash
mpirun -np 4 ./my-prover
```

The SDK detects the MPI environment automatically. You do not need to add any MPI-specific code to your host program.

### Memory considerations

Each MPI process needs its own copy of the proving key in shared memory. The proving key can be large -- several gigabytes for complex circuits. Before choosing your process count, make sure your system has enough memory:

```
Total memory needed ~= (proving key size) x (number of MPI processes) + (trace data)
```

If you run out of memory, reduce the number of processes. Two well-resourced processes will outperform four processes that are swapping to disk.

### macOS note

MPI is not fully supported on macOS. If you are developing on macOS and do not need distributed proving, use the `disable_distributed` feature flag when building:

```bash
cargo build --release --features disable_distributed
```

This compiles the prover without MPI support, avoiding linker issues on macOS. Your proofs will be generated in a single process, which is sufficient for development and testing. For production proving with MPI, use Linux.

## Coordinator-worker architecture

For workloads that exceed what a single machine can handle, ZisK provides a coordinator-worker architecture. A central coordinator receives proof requests, assigns work to a pool of workers, and aggregates the results.

### Three phases

The distributed proving process has three phases:

**1. Contributions.** The coordinator analyzes the program and assigns partition ranges to workers. Each worker receives a range of the execution trace to compute. Workers execute the guest program for their assigned range and report back the contribution data.

**2. Prove.** Workers generate partial proofs for their assigned partitions. Each worker proves its portion of the execution trace independently. This is where most of the computation happens, and where GPU acceleration (described in [GPU Proving](/optimization/gpu-proving)) makes the biggest difference.

**3. Aggregate.** The coordinator collects partial proofs from all workers and aggregates them into a single final proof. This final proof is indistinguishable from one generated on a single machine -- the verifier does not know or care that the work was distributed.

This three-phase approach is a direct consequence of ZisK's VADCOP architecture, described in [The Proving Pipeline](/how-zisk-works/proving-pipeline). Because VADCOP composes independent sub-proofs, the partial proofs from different workers can be combined without any loss of soundness.

### Running the coordinator

Start the coordinator with a configuration file:

```bash
zisk-coordinator --config coordinator.toml
```

A minimal coordinator configuration:

```toml
[coordinator]
listen_addr = "0.0.0.0:50051"
num_workers = 3
proving_key_path = "/path/to/provingKey"
```

The coordinator listens for worker connections on the specified gRPC address. It waits until `num_workers` workers have connected before accepting proof requests.

#### Coordinator options

| Option | Description |
| --- | --- |
| `listen_addr` | gRPC address to listen on |
| `num_workers` | Number of workers to wait for before starting |
| `proving_key_path` | Path to the proving key |
| `output_path` | Where to write the final proof |
| `webhook_url` | URL to notify when proof completes |

### Running workers

Start each worker with its own configuration:

```bash
zisk-worker --config worker.toml
```

A minimal worker configuration:

```toml
[worker]
coordinator_addr = "http://coordinator-host:50051"
proving_key_path = "/path/to/provingKey"
```

Workers connect to the coordinator and wait for assignments. Each worker needs access to the same proving key that the coordinator is using. In a Docker or Kubernetes deployment, this is typically a shared volume mount.

Once connected, workers are fully managed by the coordinator. They receive work assignments, compute their portions, and return results -- no manual intervention required.

### Submitting a proof request

With the coordinator running and all workers connected, submit a proof request:

```bash
zisk-coordinator prove \
  --elf path/to/elf \
  --input input.bin \
  --output /tmp/proof
```

The coordinator distributes the work, collects results, and writes the final proof to the output path. If a webhook URL is configured, it also sends a notification when the proof is ready.

## Docker deployment

For production deployments, Docker provides a consistent environment across machines. ZisK provides Dockerfiles for both CPU-only and GPU-enabled builds.

### Building the Docker image

For CPU-only proving:

```bash
docker build -t zisk-prover .
```

For GPU-enabled proving:

```bash
docker build -t zisk-prover-gpu --build-arg FEATURES=gpu .
```

### Starting the coordinator

```bash
docker run -d --name coordinator \
  -v /path/to/provingKey:/provingKey \
  -p 50051:50051 \
  zisk-prover zisk-coordinator --config /etc/zisk/coordinator.toml
```

The `-v` flag mounts the proving key into the container. The `-p` flag exposes the gRPC port so workers can connect.

### Starting workers

For CPU workers:

```bash
docker run -d --name worker1 \
  -v /path/to/provingKey:/provingKey \
  zisk-prover zisk-worker --config /etc/zisk/worker.toml
```

For GPU workers, add `--gpus all` to give the container access to the host's GPUs:

```bash
docker run -d --name worker-gpu \
  --gpus all \
  -v /path/to/provingKey:/provingKey \
  zisk-prover-gpu zisk-worker --config /etc/zisk/worker.toml
```

You can mix CPU and GPU workers in the same cluster. The coordinator assigns work without regard to worker hardware -- it is up to you to ensure that workers are provisioned appropriately for the workloads you expect.

### Scaling workers

To add more workers, start more containers pointing at the same coordinator. If you are using an orchestration system like Kubernetes, you can scale workers dynamically based on the proof queue depth.

When scaling, keep in mind:

- Each worker needs enough memory for the proving key and its portion of the execution trace.
- GPU workers should have dedicated GPUs when possible. Sharing a GPU across too many workers reduces throughput.
- The coordinator is lightweight and does not need a GPU.

## Webhook notifications

The coordinator can notify an external service when a proof completes. This is useful for integrating ZisK into larger pipelines -- for example, submitting the proof to an on-chain verifier contract.

Configure the webhook in the coordinator's configuration file:

```toml
[coordinator]
listen_addr = "0.0.0.0:50051"
num_workers = 3
proving_key_path = "/path/to/provingKey"
webhook_url = "https://your-service.com/proof-complete"
```

When a proof finishes, the coordinator sends a POST request to the webhook URL with a JSON payload containing the proof status and output path. Your service can then pick up the proof and process it however you need.

## Putting it all together

A typical production setup combines all three optimization techniques from this chapter:

1. **Precompiles** reduce the execution trace size by handling expensive operations natively.
2. **GPU proving** accelerates the polynomial math on each worker machine.
3. **Distributed proving** spreads the remaining work across multiple machines.

The combination is multiplicative. Precompiles shrink the trace, GPU acceleration speeds up each worker's proving, and distribution divides the remaining work across the cluster.
