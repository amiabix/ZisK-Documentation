---
title: Precompile Hints
description: Accelerate proof generation by pre-computing precompile results outside the VM and streaming them to the prover.
---

Precompiles execute expensive operations natively, but the prover still needs to verify each result inside the zkVM circuit. The precompile hints system accelerates this by pre-computing results outside the VM and feeding them to the prover as verifiable data through a parallel streaming pipeline.

The system has three parts:

1. **The guest emits hint requests** describing each precompile operation and its inputs.
2. **A hints processor** runs these operations in parallel outside the VM.
3. **The prover consumes the results**, verifying correctness inside the circuit without recomputing them.

::callout{icon="i-heroicons-information-circle"}
Hints are only supported in ASM mode. The emulator does not use the hints pipeline.
::

## Hint Format

Hints are transmitted as a stream of `u64` values. Each hint request has a header followed by data:

| Field | Size | Content |
| --- | --- | --- |
| Header | 1 `u64` | Hint code (upper 32 bits) and payload length in bytes (lower 32 bits) |
| Data | N `u64` values | The input to the precompile operation |

The last `u64` in the data may contain padding bytes if the payload length is not a multiple of 8.

### Control codes

Control codes manage the stream lifecycle:

| Code | Name | Description |
| --- | --- | --- |
| `0x0000` | `CTRL_START` | Reset processor state. Must be the first hint. |
| `0x0001` | `CTRL_END` | Signal end of stream. Blocks until all pending hints complete. |

A valid hint stream follows: `CTRL_START`, then data hints, then `CTRL_END`.

### Built-in hint types

| Code | Operation |
| --- | --- |
| `0x0100` | SHA-256 |
| `0x0200` | BN254 G1 add |
| `0x0201` | BN254 G1 scalar multiplication |
| `0x0205` | BN254 pairing check |
| `0x0300` | secp256k1 ECDSA address recovery |
| `0x0301` | secp256k1 ECDSA verify + address recovery |
| `0x0380` | secp256r1 (P-256) ECDSA verification |
| `0x0400` | BLS12-381 G1 add |
| `0x0401` | BLS12-381 G1 multi-scalar multiplication |
| `0x0405` | BLS12-381 G2 add |
| `0x0406` | BLS12-381 G2 multi-scalar multiplication |
| `0x040A` | BLS12-381 pairing check |
| `0x0410` | BLS12-381 map field element to G1 |
| `0x0411` | BLS12-381 map field element to G2 |
| `0x0500` | Modular exponentiation |
| `0x0600` | KZG proof verification |
| `0x0700` | Keccak-256 |

### Pass-through hints

Setting bit 31 of the hint code (e.g., `0x8000_0000 | actual_code`) marks a hint as pass-through. The data bypasses computation and is forwarded directly to the sink. This is useful for pre-computed results that do not need processing.

### Stream batching

Individual hints that exceed the transport message size limit (128 KB) are split into multiple sequential chunks. Each chunk includes a header with the total length, allowing the receiver to reassemble before processing.

## CLI Usage

Four CLI commands support hints via the `--hints` option: `execute`, `prove`, `verify-constraints`, and `stats`.

```bash
cargo-zisk prove -e path/to/elf -i input.bin --hints file://path/to/hints.bin
```

Supported URI schemes:

| Scheme | Transport |
| --- | --- |
| `file://path` | File |
| `unix://path` | Unix socket |
| `quic://host:port` | QUIC stream |
| Plain path | File (default) |

## Distributed Hints

In the coordinator-worker architecture, hints are received by the coordinator and broadcast to all workers via gRPC. The coordinator validates incoming messages, assigns sequence numbers for ordering, and dispatches them asynchronously. Workers buffer incoming messages and reorder by sequence number before processing.

### Streaming mode

The coordinator connects to a hints source and broadcasts to workers in real time:

```bash
zisk-coordinator prove \
    --hints-uri unix:///tmp/hints.sock \
    --stream-hints \
    --elf path/to/elf \
    --input input.bin
```

### Path mode

Workers load hints from a local path instead of streaming from the coordinator. Useful for debugging or when hints are pre-generated:

```bash
zisk-worker --config worker.toml --hints-uri file:///path/to/hints.bin
```

## Generating Hints in Guest Programs

To generate hints from a guest program:

1. **Emit hint requests.** Patch your code or dependent crates to call the FFI hint helper functions that generate the required input data.
2. **Add `ziskos`** to your guest `Cargo.toml`.
3. **Initialize and finalize the hint stream.** Call the init and close functions before and after the precompile code.
4. **Compile with `--cfg zisk_hints`.** This activates hint code generation for the native target.
5. **Ensure deterministic execution.** The native execution that generates hints must produce them in the exact same order as the guest compiled for the `zkvm/zisk` target expects to consume them.

### Initialize and finalize

Start hint generation with one of:

```rust
use ziskos::hints::{init_hints_file, init_hints_socket, close_hints};

// Write hints to a file
init_hints_file(PathBuf::from("/tmp/hints.bin"), None)?;

// Or stream hints through a Unix socket
init_hints_socket(PathBuf::from("/tmp/hints.sock"), None, None)?;
```

Close hint generation when done:

```rust
close_hints()?;
```

Guard these calls with the `zisk_hints` cfg flag so they only run during native hint-generation builds:

```rust
#[cfg(zisk_hints)]
{
    init_hints_file(PathBuf::from("/tmp/hints.bin"), None)?;
}

// ... precompile code ...

#[cfg(zisk_hints)]
{
    close_hints()?;
}
```

### Emit hint requests

Call the FFI helper functions from the `ziskos` crate to emit hints. For example, for SHA-256:

```rust
#[cfg(zisk_hints)]
unsafe {
    hint_sha256(input.as_ptr(), input.len());
}
```

This generates the hint input data using the same values that the zkVM will use when executing the `zkvm/zisk` target. The hints processor later runs the operation and feeds the result to the prover.

| Code | Function |

| `0x0205` | `hint_bn254_pairing_check(pairs: *const u8, num_pairs: usize)` |
| `0x0300` | `hint_secp256k1_ecdsa_address_recover(sig: *const u8, recid: *const u8, msg: *const u8)` |
| `0x0301` | `hint_secp256k1_ecdsa_verify_and_address_recover(sig: *const u8, msg: *const u8, pk: *const u8)` |
| `0x0380` | `hint_secp256r1_ecdsa_verify(msg: *const u8, sig: *const u8, pk: *const u8)` |

### Compile for hint generation

```bash
RUSTFLAGS='--cfg zisk_hints' cargo build --release
```

To also print hint metrics during execution:

```bash
RUSTFLAGS='--cfg zisk_hints --cfg zisk_hints_metrics' cargo build --release
```

To enable hint support when building the ELF for the zkVM, pass `--hints` to `cargo-zisk rom-setup`.

### Deterministic execution

The native execution that generates hints and the guest compiled for `zkvm/zisk` must produce and consume hints in the exact same order. To guarantee this:

- Do not use threads or parallel execution in code paths that emit hints.
- Do not iterate over `HashMap` or other non-deterministically ordered data structures in code paths that emit hints.

Any divergence in hint ordering between native and zkVM execution will cause incorrect behavior.

## Custom Hint Handlers

Register custom handlers via the `HintsProcessor` builder:

```rust
let processor = HintsProcessor::builder(my_sink)
    .custom_hint(0xA000, |data: &[u64]| -> Result<Vec<u64>> {
        Ok(vec![data[0] * 2])
    })
    .custom_hint(0xA001, |data| {
        Ok(transform(data))
    })
    .build()?;
```

Requirements:
- Handler function must be `Fn(&[u64]) -> Result<Vec<u64>> + Send + Sync + 'static`.
- Use codes in the range `0xA000`-`0xFFFF` to avoid conflicts with built-in types.

To emit custom hints from the guest, use:

```rust
hint_custom(hint_id: u32, data_ptr: *const u8, data_len: usize, is_result: u8);
```
