What happens inside the prover: how ZisK takes a guest program and its input and produces a proof.

## Three Modes of Running a Guest

The `ProverClient` offers three ways to run a guest, each doing more work than the last:

**Execute** runs the program and returns the output. No proof. Fastest mode; use during development.

**Verify constraints** runs the program, builds the execution trace, and checks every PIL constraint. No proof, but catches bugs that execution misses (e.g., a precompile that produces the right answer but violates an internal consistency rule). If constraints pass, proving will succeed.

**Prove** runs the program, builds the trace, and generates a proof. Slowest, but the only mode that produces a verifiable result.

Development cycle: execute while building, verify constraints when ready, prove when you need a proof.

## The Execution Trace

When ZisK runs your guest for proving (or constraint verification), it records everything: every instruction, register value, and memory access. This record is the **execution trace**.

The trace is essentially a table. Each row is one step; columns hold the instruction, registers, and memory state. Simple programs produce thousands of rows, complex ones can produce billions.

```
Step │ PC     │ Instruction │ Registers...        │ Memory...
─────┼────────┼─────────────┼─────────────────────┼──────────
   0 │ 0x1000 │ addi x1, 5  │ x1=5, x2=0, ...    │ ...
   1 │ 0x1004 │ add x2, x1  │ x1=5, x2=5, ...    │ ...
   2 │ 0x1008 │ sw x2, 0(sp)│ x1=5, x2=5, ...    │ [sp]=5
   3 │ 0x100c │ ...         │ ...                 │ ...
  ...│ ...    │ ...         │ ...                 │ ...
```

The prover's job is to convince a verifier that this trace is valid: that it represents a real execution that started with the claimed input and produced the claimed output.

## PIL Constraints

ZisK defines trace validity through **PIL** (Polynomial Identity Language), which expresses the rules of the RISC-V ISA as polynomial equations. Each constraint must hold for every row (or between specific rows). For example:

- If the current instruction is `add x3, x1, x2`, then the value in the `x3` column of the next row must equal the sum of the `x1` and `x2` columns of the current row.
- The program counter must advance correctly, either by 4 (sequential execution) or to the branch target (if a branch was taken).
- Memory reads must return the last value written to that address.

Hundreds of constraints collectively encode the full RISC-V ISA plus precompile operations. If the program ran correctly, all constraints hold. Tampering with the trace (e.g., changing the output without re-running) violates at least one.

Constraint verification evaluates every constraint on every row. It catches bugs that execution misses because it checks intermediate consistency, not just the final output.

## The Proving Pipeline

Proof generation is a multi-stage pipeline, each stage trading proof size for verification cost.

## Stage 1: VADCOP STARK

The first and most fundamental stage produces a STARK proof using **VADCOP** (Verifiable Arithmetic Deterministic Computation Over Polynomials).

The prover takes the execution trace, interpolates it into polynomials, and uses the FRI (Fast Reed-Solomon Interactive Oracle Proofs of Proximity) protocol to prove that those polynomials satisfy all PIL constraints. The result is a STARK proof.

STARKs require no trusted setup, are post-quantum secure (hash-based, no elliptic curve assumptions), and scale well with computation size. The trade-off is proof size: too large for on-chain use, but fine for off-chain verification or as a pipeline intermediate.

`.prove().run()` produces a VADCOP STARK by default.

## Stage 2: Compressed Proof

The compressed stage wraps the STARK in a recursive proof: instead of verifying the large proof directly, you prove *that you verified it*. Since verification is a smaller computation than the original program, the outer proof is smaller.

The result is meaningfully smaller than the raw STARK but still too large for cheap on-chain verification. Useful for proof aggregation or size-sensitive off-chain workflows.

## Stage 3: PLONK SNARK

The final stage wraps the compressed proof in a **PLONK SNARK**, a succinct proof system based on elliptic curves. The result is a small constant-size proof, verifiable on-chain in a single transaction.

Proof type is selected through the prove builder:

```rust
// VADCOP STARK (default)
let result = client.prove(&pk, stdin).run()?;

// Compressed STARK
let result = client.prove(&pk, stdin).compressed().run()?;

// PLONK SNARK (requires .snark() on the ProverClient builder)
let result = client.prove(&pk, stdin).plonk().run()?;
```

## What the Proof Contains

A proof makes a precise claim: *this program, identified by its verification key, was executed and produced these public values.* It does not contain the input, the execution trace, or the program. Only the mathematical commitments needed for verification.

Verifiers check the math. No re-execution, no trust in the prover, no knowledge of the input required.

## Proving and Verification Keys

`.setup(elf)` analyzes the guest ELF and generates two keys:

- **Proving key (pk)**: everything the prover needs. Encodes the constraint system, polynomial commitments, and proof parameters. Large (often hundreds of MB). Prover-only.
- **Verification key (vk)**: compact commitment to the program. Small enough to embed in a smart contract. Verifier-only.

The vk is derived from the guest program. Different program, different vk. A proof is bound to its vk and cannot be passed off as coming from a different program.

```rust
let (pk, vk) = client.setup(&ELF)?;

// Generate a proof using the proving key
let result = client.prove(&pk, stdin).run()?;

// Verify the proof using the verification key
client.verify(result.get_proof(), result.get_publics(), &vk)?;
```

Setup is expensive. In production, generate keys once and reuse for multiple proofs of the same program. Keys only change when the guest changes.
