How data moves through a ZisK program: host writes input, guest reads and computes, guest commits output, host reads the result.

## The Big Picture

Data flows in one direction at a time. The host writes all input before execution begins. The guest reads that input during execution, computes, and commits its output. After execution, the host reads the committed output. There is no interactive communication: no back-and-forth, no callbacks, no shared memory.

This one-way model ensures determinism. If the guest could call back to the host, different hosts could provide different responses, breaking reproducibility.

## Sending Input: Host to Guest

On the host side, you build up an input buffer using `ZiskStdin`:

```rust
let stdin = ZiskStdin::new();
stdin.write(&10u32);
stdin.write(&"hello".to_string());
stdin.write(&vec![1u8, 2, 3]);
```

Each call to `.write(&value)` serializes the value using [bincode](https://github.com/bincode-org/bincode) and appends the bytes to the buffer. You can call `.write()` as many times as you like; values are appended in order, one after another.

On the guest side, each call to `io::read::<T>()` deserializes the next value from that buffer:

```rust
let n: u32 = io::read();           // reads the 10u32
let msg: String = io::read();       // reads "hello"
let bytes: Vec<u8> = io::read();    // reads [1, 2, 3]
```

The two sides must agree on types and order. If the host writes a `u32` followed by a `String`, the guest must read a `u32` followed by a `String`, in that order. If the types don't match, deserialization will fail and the guest will panic.

## Serialization Details

ZisK uses bincode for all serialization between host and guest. Bincode is a compact binary format: no field names, no delimiters, just raw data. This keeps the input buffer small, which matters because the entire input becomes part of the execution trace.

Here are the sizes of common types:

| Type | Bincode size |
| --- | --- |
| `u8` / `i8` / `bool` | 1 byte |
| `u32` / `i32` | 4 bytes |
| `u64` / `i64` | 8 bytes |
| `u128` / `i128` | 16 bytes |
| `String` / `Vec<T>` | 8 bytes (length as `u64`) + data |

Integers are encoded in little-endian byte order. Strings and vectors are length-prefixed: bincode writes the length as a `u64` (8 bytes), then the raw contents. Structs are serialized field by field in declaration order; there are no field names in the binary representation.

For your own structs, derive `Serialize` and `Deserialize` from serde:

```rust
use serde::{Serialize, Deserialize};

#[derive(Serialize, Deserialize)]
struct FibInput {
    n: u32,
    label: String,
}
```

Both guest and host must agree on the struct definition. Put shared types in a common crate that both depend on.

## Reading Input: The Guest Side

Inside the guest, `io::read::<T>()` consumes bytes from the input buffer and deserializes them into the requested type. The buffer is read sequentially; there is no way to skip ahead, seek backward, or peek at what comes next.

If you try to read past the end of the input; for example, the host wrote two values but the guest tries to read three, the guest will panic. This is a hard failure, not an error you can recover from. The input must contain exactly what the guest expects.

A common pattern is to write a single struct that contains all the input the guest needs:

```rust
// Shared type (in a common crate)
#[derive(Serialize, Deserialize)]
struct ProgramInput {
    iterations: u32,
    seed: [u8; 32],
    data: Vec<u8>,
}

// Host
let input = ProgramInput { iterations: 100, seed: [0u8; 32], data: payload };
stdin.write(&input);

// Guest
let input: ProgramInput = io::read();
```

This avoids ordering mistakes. Instead of writing and reading five separate values in the right sequence, you write and read one struct.

## Committing Output: Guest to Proof

`io::commit(&value)` serializes the value with bincode and writes it to the **public output region**. This data becomes part of the proof. Verifiers can read it and confirm it came from the proven computation.

```rust
// Inside the guest
io::commit(&n);       // "I computed Fibonacci for n = 10"
io::commit(&result);  // "The answer is 89"
```

You can call `io::commit()` multiple times. Each call appends to the same output region. The values are written sequentially, just like input.

**The public output region is 256 bytes** (64 `u32` slots). Exceeding this fails proving. For large outputs, commit a hash and provide the full data alongside the proof.

## Commit vs. Write

The guest has two functions for writing output, both targeting the same public output region:

**`io::commit(&value)`** serializes a value with bincode and writes the resulting bytes to the proof's public output region. Use this for typed data.

**`io::write(buf)`** writes raw bytes directly to the same public output region. `io::commit()` calls `io::write()` internally after serializing.

Both are subject to the 256-byte public output limit. Use `io::commit()` for structured data; use `io::write()` only when you need to write pre-serialized or raw bytes.

## Reading Output: The Host Side

After the guest finishes executing (or after proof generation), the host reads the committed output:

```rust
let client = ProverClient::builder().emu().build().unwrap();
let (pk, vk) = client.setup(&ELF)?;

let result = client.execute(&pk, stdin)?;

// Read typed public values
let output: MyOutput = result.get_public_values()?;
println!("fib({}) = {}", output.n, output.value);
```

`.get_public_values::<T>()` deserializes the committed values. The type must match what the guest committed.

After proving:

```rust
let result = client.prove(&pk, stdin).run()?;

// Read public values from the proof result
let output: MyOutput = result.get_public_values()?;
```

## The Tape Model

Both input and output are sequential tapes: read/write forward only, no random access, no seeking.

```
Input tape:   [  n  |  data  |  config  ]
                 ^
              read head (moves right with each io::read())

Output tape:  [  hash  |  count  ]
                          ^
              write head (moves right with each io::commit())
```

The order of `write()` and `read()` calls is a contract between host and guest. Reorder one side, reorder the other to match.
