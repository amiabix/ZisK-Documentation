Every ZisK project has a two-crate structure: a `guest` and a `host`. This split is fundamental to how provable computation works.

## The Two-Program Model

ZisK programs have two halves that serve fundamentally different purposes.

The **guest** runs inside the ZisK virtual machine. It is the program you are proving, the computation whose correctness a verifier will check. Because it runs inside a VM, it is deterministic and isolated. Given the same input, it always produces the same output. It cannot access the filesystem, the network, or anything outside the VM. This isolation is what makes it provable.

The **host** runs on your machine, just like any other Rust program. It is the orchestrator: it reads files, calls APIs, makes decisions, assembles input for the guest, invokes the prover, and handles the resulting proof. The host is unconstrained; it can do whatever a normal program can do.

The two halves communicate through a well-defined interface: the host writes input before execution begins, the guest reads that input during execution, and the guest commits output that the host can read after execution completes. There is no back-and-forth during execution; the guest runs to completion on whatever input it was given.

Provable computation requires determinism. If the guest could make network calls or read the clock, different runs could produce different results, and the proof would be meaningless. VM isolation guarantees reproducibility.

## The Guest Program

The Fibonacci guest from the previous chapter:

```rust
#![no_main]
ziskos::entrypoint!(main);

use ziskos::io;
use serde::{Serialize, Deserialize};

#[derive(Serialize, Deserialize, Debug)]
struct FibResult {
    n: u32,
    value: u64,
}

fn main() {
    let n: u32 = io::read();

    let mut a: u64 = 0;
    let mut b: u64 = 1;
    for _ in 0..n {
        let temp = b;
        b = a + b;
        a = temp;
    }

    let result = FibResult { n, value: a };
    io::commit(&result);
}
```

The ZisK-specific parts:

**`#![no_main]`**: no standard entry point. The VM provides the runtime; there's no OS, no `libc`, no argument parsing.

**`ziskos::entrypoint!(main)`**: registers the VM entry point.

**`ziskos::io`**: the guest's I/O interface. Three functions:

- **`io::read::<T>()`** reads the next value from the host's input stream. The value is deserialized using bincode. The host must have written a value of the matching type.

- **`io::commit(&value)`** serializes a value with bincode and writes the bytes to the proof's public output region. This data becomes part of the proof itself; anyone who verifies the proof can read it. This is how you make results publicly verifiable.

- **`io::write(buf)`** writes raw bytes to the same public output region as `io::commit()`, but without bincode serialization. `io::commit()` calls `io::write()` internally after serializing. Use `io::commit()` for typed data; use `io::write()` only when you need to write pre-serialized or raw bytes.

Beyond I/O, the guest is ordinary Rust. `std` is available (`Vec`, `HashMap`, `String`, etc.). Third-party crates work as long as they compile to RISC-V. Most do. `println!` works in the emulator; it's stripped during proving.

No filesystem, no network, no randomness, no clock. Pure computation.

## The Host Program

The host is a regular Rust binary. All host programs follow the same pattern:

1. Build the input using `ZiskStdin`
2. Create a `ProverClient`
3. Run the guest (either execute or prove)
4. Read the results

The host uses a `build.rs` script to compile the guest at build time. When you run `cargo build` on the host crate, the build script invokes the ZisK toolchain to compile the guest to a RISC-V ELF binary. The `include_elf!` macro then embeds that binary into the host, so the host always carries a copy of the exact guest program it will run.

The guest ELF is compiled once and embedded. If you change guest code, rebuild the host to pick up the new ELF.

The host can do anything a normal Rust program can. The only constraint is that input to the guest must be serializable with bincode.

## The Virtual Machine

ZisK compiles guest programs to RISC-V, specifically the `riscv64ima-zisk-zkvm-elf` target. This is a 64-bit RISC-V target with integer multiplication/division and atomic extensions, running on ZisK's bare-metal operating system (ZiskOS).

When you execute or prove a guest, the VM runs those RISC-V instructions and records every step. Every instruction executed, every register read, every memory access; all of it becomes the **execution trace**, which is the raw material for proof generation.

ZisK provides two execution backends:

**Emulator** (`.emu()`). A portable interpreter. Works on all platforms (macOS, Linux, any architecture). Use during development.

**ASM** (`.asm()`). A native x86_64 backend, Linux only. Translates RISC-V to native machine code. Significantly faster. Use for production proving.

**Precompiles.** Expensive operations (SHA-256, Keccak-256, elliptic curve arithmetic) are implemented as native VM instructions rather than interpreted instruction-by-instruction. This dramatically reduces execution trace size. See the [Optimization](/optimization/precompiles) chapter.

## Project Structure

Every ZisK project follows this workspace layout:

```
fibonacci/
├── Cargo.toml
├── guest/
│   ├── Cargo.toml
│   └── src/main.rs
└── host/
    ├── Cargo.toml
    ├── build.rs
    ├── src/main.rs
    └── bin/
        ├── execute.rs
        ├── prove.rs
        ├── compressed.rs
        ├── plonk.rs
        ├── verify-constraints.rs
        └── ziskemu.rs
```

The guest crate compiles to RISC-V. The host crate depends on the ZisK SDK and uses `build.rs` to trigger guest compilation. `bin/` contains binaries for different workflows. Add more guest crates or host binaries as needed.
