title: GPU Proving
description: Use NVIDIA GPUs to accelerate proof generation.

The most computationally expensive part of proof generation is the polynomial math -- computing FFTs, evaluating constraints, and building commitment trees. These operations are highly parallel and map well onto GPU hardware. ZisK supports offloading this work to NVIDIA GPUs.

## Requirements

GPU proving requires:

- **Linux x86_64** -- GPU proving is not supported on macOS or Windows
- **NVIDIA GPU** with CUDA compute capability 7.0 or higher
- **CUDA toolkit** installed and available on your PATH

If you installed ZisK using the standard installer (as described in [Install on Linux](/getting-started/install-linux)), the resulting binaries do not include GPU support by default. You need to build from source with the GPU feature enabled.

## Building with GPU support

When building ZisK from source, enable the `gpu` feature:

```bash
cargo build --release --features gpu
```

It is recommended to compile ZisK directly on the server where it will be executed. The binary will be optimized for the local GPU architecture, which can lead to better runtime performance.

This compiles the CUDA kernels and links them into the prover. The build process requires the CUDA toolkit, so make sure `nvcc` is available:

```bash
nvcc --version
```

## Using GPU proving from the SDK

To enable GPU proving in your host program, add `.gpu()` to the prover builder chain:

```rust
use anyhow::Result;
use zisk_sdk::{ProverClient, ParamsGPU};

fn main() -> Result<()> {
    let client = ProverClient::builder()
        .emu()
        .gpu(Some(ParamsGPU::default()))
        .build()
        .unwrap();

    // Use the prover as normal, proving will use the GPU
    let (pk, vk) = client.setup(&ELF)?;
    let result = client.prove(&pk, stdin).run()?;

    Ok(())
}
```

The `.gpu()` call tells the prover to offload polynomial operations to the GPU. Everything else -- emulation, trace generation, proof aggregation -- still runs on the CPU.

## GPU options

For fine-grained control over GPU behavior, use `ParamsGPU`:

The `.gpu()` method accepts an `Option<ParamsGPU>`. Passing `Some(ParamsGPU::default())` lets the GPU driver manage resource allocation automatically. In most cases, this is what you want. The GPU will use as many streams and as much memory as it determines is optimal for your hardware.

## GPU proving from the CLI

GPU proving is enabled automatically when ZisK is built with the `gpu` feature. There is no separate CLI flag. If your `cargo-zisk` binary was built with `--features gpu`, the prove command will use GPU acceleration:

```bash
cargo-zisk prove -e path/to/elf -i input.bin -o /tmp/proof
```

For more CLI options, see the [CLI Reference](/cli-reference).

## Combining GPU with MPI

GPU proving and MPI parallelism are complementary. MPI splits the proving workload across multiple processes, and each process can use a GPU for its portion of the work. If your machine has multiple GPUs, this lets you use all of them:

```bash
mpirun -np 4 cargo-zisk prove -e path/to/elf -i input.bin -o /tmp/proof
```

Each MPI process will attempt to use the GPU. If you have fewer GPUs than processes, the processes will share GPU resources. For best results, match the number of MPI processes to the number of available GPUs.

::callout{icon="i-heroicons-exclamation-triangle"}
GPU memory is typically more limited than CPU memory. When combining GPU execution with concurrent proof generation, ensure that each process has sufficient memory available on the GPU to avoid out-of-memory errors.
::

See [Distributed Proving](/optimization/distributed-proving) for more on MPI.

## Troubleshooting

**"CUDA not found" during build.** Make sure the CUDA toolkit is installed and `nvcc` is on your PATH. On most systems, the toolkit installs to `/usr/local/cuda`, and you may need to add `/usr/local/cuda/bin` to your PATH.

**Out-of-memory errors during proving.** Large execution traces require significant GPU memory. If you run out of VRAM, try reducing the number of MPI processes sharing the GPU, or use a GPU with more memory.

**No speedup observed.** GPU acceleration benefits the polynomial math phase. If your program has a very small execution trace, the overhead of transferring data to the GPU may outweigh the speedup. GPU proving is most effective for large traces.
