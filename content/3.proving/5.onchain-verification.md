## Prerequisites

- A PLONK SNARK proof (see [Proof Types](/proving/proof-types))
- [Foundry](https://getfoundry.sh/) installed
- An RPC endpoint and funded account on your target chain

## The Verifier Interface

The ZisK verifier contract exposes two functions:

```solidity
interface IZiskVerifier {
    function verifySnarkProof(
        uint64[4] calldata programVK,
        uint64[4] calldata rootCVadcopFinal,
        bytes calldata publicValues,
        bytes calldata proofBytes
    ) external view;

    function getRootCVadcopFinal() external pure returns (uint64[4] memory);
}
```

**`verifySnarkProof`** takes four parameters:

- **`programVK`**: fingerprint of the guest program. Different programs produce different values.
- **`rootCVadcopFinal`**: commitment to the VADCOP circuit. Fixed per ZisK version.
- **`publicValues`**: the outputs the guest committed, encoded as bytes.
- **`proofBytes`**: the raw PLONK proof.

The function does not return a value. It either succeeds silently or reverts with `InvalidProof()`.

**`getRootCVadcopFinal`** returns the expected `rootCVadcopFinal` for this verifier deployment. Since this value is fixed per ZisK version, the contract embeds it as a constant. Use this to query the correct value when submitting proofs.

## How Verification Works Internally

The contract computes a public values digest from the three data parameters and verifies the PLONK proof against it:

```solidity
uint256 publicValuesDigest = hashPublicValues(programVK, rootCVadcopFinal, publicValues);
```

The `hashPublicValues` function builds a preimage by packing the components in this order:

```
programVK (32 bytes) | publicValues (variable) | rootCVadcopFinal (32 bytes)
```

Each `uint64[4]` array is packed as four `bytes8` values (32 bytes total). `publicValues` is included as-is between them.

The SHA-256 hash of this preimage is then reduced modulo the BN254 scalar field:

```solidity
uint256(sha256(preimage)) % _RFIELD
```

where `_RFIELD` is `21888242871839275222246405745257275088548364400416034343698204186575808495617`.

::callout{icon="i-heroicons-exclamation-triangle"}
The preimage order is `programVK | publicValues | rootCVadcopFinal`. Getting this order wrong is a common source of verification failure.
::

## Byte Order

ZisK stores values in little-endian internally. Solidity and the EVM use big-endian. Every `uint64` and `uint32` must be converted with `.to_be_bytes()` before passing to the contract.

The SDK provides helper methods that handle this conversion:

- **`publics.public_bytes_solidity()`** converts the 64 public `u32` values from LE to BE, producing 256 bytes.
- **`publics.bytes_solidity(&program_vk, &vadcop_verkey)`** builds the full hash preimage with all byte swaps applied.

::callout{icon="i-heroicons-exclamation-triangle"}
Getting the endianness wrong is the single most common cause of on-chain verification failure. If your proof fails to verify on-chain but passes local verification, check the byte order first.
::

## Getting the Values from Rust

Generate a PLONK proof and extract the components:

```rust
use anyhow::Result;
use zisk_sdk::{include_elf, ElfBinary, ProverClient, ZiskIO, ZiskProof, ZiskStdin};

const ELF: ElfBinary = include_elf!("guest");

fn main() -> Result<()> {
    let client = ProverClient::builder()
        .emu()
        .snark()
        .build()
        .unwrap();
    let (pk, vk) = client.setup(&ELF)?;

    let stdin = ZiskStdin::new();
    stdin.write(&42u64);

    let result = client.prove(&pk, stdin)
        .plonk()
        .run()?;

    // Verify locally first
    client.verify(result.get_proof(), result.get_publics(), &vk)?;

    // Extract components for on-chain submission
    let program_vk = result.get_program_vk();  // ZiskProgramVK
    let publics = result.get_publics();         // ZiskPublics

    // Get proof bytes (PLONK variant)
    let proof = result.get_proof();
    let proof_bytes = match proof {
        ZiskProof::Plonk(bytes) => bytes,
        _ => panic!("Expected PLONK proof"),
    };

    // Get public values in Solidity byte order (LE to BE)
    let public_values_be = publics.public_bytes_solidity();

    // program_vk.vk contains the raw verification key bytes (32 bytes, 4 x u64 LE)
    // Convert each u64 from LE to BE for the contract call
    println!("Proof size: {} bytes", proof_bytes.len());
    println!("Public values: {} bytes", public_values_be.len());
    println!("Program VK: {} bytes", program_vk.vk.len());

    Ok(())
}
```

## Deploying the Verifier

Deploy the pre-built verifier from the ZisK repository using Foundry:

```bash
forge create --rpc-url $RPC_URL \
    --private-key $PRIVATE_KEY \
    src/ZiskVerifier.sol:ZiskVerifier
```

The verifier is stateless and immutable. Deploy once, reuse for all programs. The `programVK` parameter distinguishes different guest programs.

## Calling the Verifier from Your Contract

Call the verifier from a contract that takes action based on verified outputs:

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

interface IZiskVerifier {
    function verifySnarkProof(
        uint64[4] calldata programVK,
        uint64[4] calldata rootCVadcopFinal,
        bytes calldata publicValues,
        bytes calldata proofBytes
    ) external view;

    function getRootCVadcopFinal() external pure returns (uint64[4] memory);
}

contract MyVerifiedApp {
    IZiskVerifier public immutable verifier;

    event ComputationVerified(uint256 result);

    constructor(address _verifier) {
        verifier = IZiskVerifier(_verifier);
    }

    function submitProof(
        uint64[4] calldata programVK,
        bytes calldata publicValues,
        bytes calldata proofBytes,
        uint256 claimedResult
    ) external {
        // Query the rootCVadcopFinal from the verifier contract
        uint64[4] memory rootC = verifier.getRootCVadcopFinal();

        // Verify the ZisK proof (reverts if invalid)
        verifier.verifySnarkProof(programVK, rootC, publicValues, proofBytes);

        // Proof is valid. Your application logic goes here.
        emit ComputationVerified(claimedResult);
    }
}
```

The `getRootCVadcopFinal()` call retrieves the expected value from the verifier contract, so your application does not need to hardcode it.

## End-to-End Example

Full flow from Rust to Solidity:

**Step 1: Generate the PLONK proof in Rust**

```rust
use anyhow::Result;
use zisk_sdk::{include_elf, ElfBinary, ProverClient, ZiskIO, ZiskStdin};

const ELF: ElfBinary = include_elf!("guest");

fn main() -> Result<()> {
    let client = ProverClient::builder()
        .emu()
        .snark()
        .build()
        .unwrap();
    let (pk, vk) = client.setup(&ELF)?;

    let stdin = ZiskStdin::new();
    stdin.write(&42u64);

    let result = client.prove(&pk, stdin)
        .plonk()
        .run()?;

    // Verify locally first
    client.verify(result.get_proof(), result.get_publics(), &vk)?;

    // Save the proof for submission
    result.save_proof_with_publics("/tmp/my-plonk-proof")?;

    println!("Proof generated and saved");

    Ok(())
}
```

**Step 2: Deploy the verifier**

```bash
forge create --rpc-url $RPC_URL \
    --private-key $PRIVATE_KEY \
    src/ZiskVerifier.sol:ZiskVerifier
```

**Step 3: Deploy your application contract**

```bash
forge create --rpc-url $RPC_URL \
    --private-key $PRIVATE_KEY \
    src/MyVerifiedApp.sol:MyVerifiedApp \
    --constructor-args $VERIFIER_ADDRESS
```

**Step 4: Submit the proof**

Use `cast` or ethers.js to call your contract with the programVK, publicValues, and proof bytes. Query `getRootCVadcopFinal()` from the verifier contract to get the rootCVadcopFinal value.

## Gas Costs

Gas cost is dominated by EC pairing operations. Constant cost regardless of guest program complexity.
