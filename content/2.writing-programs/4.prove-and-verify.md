So far you've only executed the guest. Execution confirms the program works, but it doesn't produce a proof. A proof lets anyone verify the computation ran correctly without re-running it. The examples here continue from [Build and Run](/writing-programs/build-and-run).

## Adding proving to main.rs

In the previous section, `host/src/main.rs` executed the guest and read back the result. Now we'll extend it to generate a proof and verify it. Update `host/src/main.rs`:

```rust
use anyhow::Result;
use serde::{Deserialize, Serialize};
use zisk_sdk::{
    ZiskStdin, ZiskIO, ElfBinary,
    ProofOpts, ProverClient, include_elf,
};

pub const ELF: ElfBinary = include_elf!("guest");

#[derive(Serialize, Deserialize, Debug)]
struct FibResult {
    n: u32,
    value: u64,
}

fn main() -> Result<()> {
    let n = 10u32;
    let stdin = ZiskStdin::new();
    stdin.write(&n);

    let client = ProverClient::builder().emu().build().unwrap();
    let (pk, _) = client.setup(&ELF)?;

    // Execute first to verify the program works
    let result = client.execute(&pk, stdin.clone())?;
    println!("Cycles: {}", result.get_execution_steps());

    let output: FibResult = result.get_public_values()?;
    println!("Fibonacci({}) = {}", output.n, output.value);

    // Generate a proof
    let stdin = ZiskStdin::new();
    stdin.write(&n);

    let proof_opts = ProofOpts::default().minimal_memory();
    let result = client.prove(&pk, stdin)
        .with_proof_options(proof_opts)
        .run()?;

    println!("Execution steps: {}", result.get_execution_steps());

    client.verify(result.get_proof(), result.get_publics(), result.get_program_vk())?;
    println!("Proof verified!");

    let fib: FibResult = result.get_public_values()?;
    println!("Proven result: Fibonacci({}) = {}", fib.n, fib.value);

    Ok(())
}
```

```bash
cargo run --release
```

You should see something like:

```
<<< EXECUTE (156ms)
<<< CALCULATING_CONTRIBUTIONS (25979ms)
<<< GENERATING_INNER_PROOFS (117966ms)
<<< GENERATE_VADCOP_FINAL_PROOF (3433ms)
Execution steps: 5255
Proof verified!
Proven result: Fibonacci(10) = 55
```

The key additions: `ProofOpts` to configure proving behavior, `client.prove()` to generate the proof, and `client.verify()` to check it.

## Setting up the client

All proving goes through `ProverClient`. You configure it with a builder and then call `.build()`:

```rust
let client = ProverClient::builder()
    .emu()       // backend: .emu() for development, .asm() for production (Linux)
    .build()
    .unwrap();

let (pk, _) = client.setup(&ELF)?;
```

The builder supports backends (`.emu()`, `.asm()`), modes (`.prove()`, `.verify_constraints()`), and optional features like `.snark()` for PLONK proof generation and `.gpu()` for GPU acceleration. `ProverClient` is a singleton: one per process, reuse it with different `.setup()` calls for multiple programs.

For all builder options, see the [ProverClient Reference](/reference/prover-client).

---

## The ProveBuilder

`client.prove(&pk, stdin)` doesn't actually prove anything. It returns a `ProveBuilder` that you configure and then call `.run()` on:

```rust
let result = client.prove(&pk, stdin)
    .with_proof_options(proof_opts)   // optional: configure proving behavior
    .run()?;                          // this is where proving actually happens
```

The builder is lazy. Nothing runs until `.run()`.

You can also chain a proof type selector before `.run()`:

```rust
// Compressed STARK
let result = client.prove(&pk, stdin).compressed().run()?;

// PLONK SNARK (requires .snark() at client build time)
let result = client.prove(&pk, stdin).plonk().run()?;
```

These proof types are covered in detail on the [Proof Types](/writing-programs/proof-types) page.

## Verification

Always verify after proving. It catches configuration issues before you ship a bad proof.

```rust
client.verify(result.get_proof(), result.get_publics(), result.get_program_vk())?;
```

`verify()` takes three references: the proof, the public values, and a verification key. All three are available directly from the prove result.

If you need to verify a proof loaded from disk, you can get the verification key from the ELF without calling `setup()` again:

```rust
let vk = client.vk(&ELF)?;
```

## Verify constraints

Before committing to a full prove run (which can take minutes), you can check that the execution trace satisfies all PIL constraints. This catches constraint violations that plain execution would miss, and it's much faster than generating an actual proof.

Build the client with `.verify_constraints()` mode, then call `verify_constraints(&pk, stdin)`:

```rust
let client = ProverClient::builder()
    .emu()
    .verify_constraints()
    .build()
    .unwrap();

let (pk, _) = client.setup(&ELF)?;

let stdin = ZiskStdin::new();
stdin.write(&10u32);

let result = client.verify_constraints(&pk, stdin)?;
println!("Cycles: {}", result.get_execution_steps());
println!("Duration: {:?}", result.get_duration());

let fib: FibResult = result.get_public_values()?;
println!("Fibonacci({}) = {}", fib.n, fib.value);
```

The result gives you the same methods as execute and prove results: `get_public_values::<T>()`, `get_execution_steps()`, and `get_duration()`. If any constraint is violated, the call returns an error.

## ProofOpts

`ProofOpts` controls proving behavior:

```rust
use zisk_sdk::ProofOpts;

let opts = ProofOpts::default().minimal_memory();

let result = client.prove(&pk, stdin)
    .with_proof_options(opts)
    .run()?;
```

Available methods: `.minimal_memory()`, `.save_proofs()`, `.verify_proofs()`, `.no_aggregation()`, `.output_dir(path)`. The defaults work for most cases. `.minimal_memory()` is recommended unless you have a specific reason to skip it.

## Saving and loading proofs

Save a proof (with or without public values) for later use:

```rust
// Save proof + publics together
result.save_proof_with_publics("tmp/proof_with_publics.bin")?;

// Save proof alone
result.get_proof().save("tmp/proof.bin")?;
```

Load and verify from disk:

```rust
use zisk_sdk::{ZiskProof, ZiskPublics, ZiskProofWithPublicValues};

// Load proof + publics together
let loaded = ZiskProofWithPublicValues::load("tmp/proof_with_publics.bin")?;
let vk = client.vk(&ELF)?;
client.verify(&loaded.proof, &loaded.publics, &vk)?;

// Load proof alone (you need to reconstruct publics separately)
let proof = ZiskProof::load("tmp/proof.bin")?;
let publics = ZiskPublics::write(&output)?;
client.verify(&proof, &publics, &vk)?;
```

`client.vk(&ELF)` derives the verification key from the ELF. Use this when you don't have the VK from `setup()`.
