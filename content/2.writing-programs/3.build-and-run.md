With the guest program written, the next step is setting up the host side: the build script that compiles the guest into a RISC-V ELF, and the host program that feeds it input and runs it. This page continues from [Project Setup](/writing-programs/project-setup).

---

## Host dependencies: `host/Cargo.toml`

The template generates this for you. Here's what's in it:

```toml
[package]
name = "host"
version = "0.1.0"
edition = "2024"
default-run = "host"

[dependencies]
zisk-sdk = { workspace = true }
anyhow = "1.0"
serde = { version = "1.0", default-features = false, features = ["derive"] }

[build-dependencies]
zisk-sdk = { workspace = true }
```

- **`zisk-sdk`** appears twice: under `[dependencies]` for the host code, and under `[build-dependencies]` for the `build.rs` script that compiles the guest
- **`default-run = "host"`** tells Cargo that `cargo run` should run `src/main.rs` by default

---

## The build script

The host crate contains a `build.rs` script that runs at compile time. It cross-compiles the guest to a RISC-V ELF and makes it available to the host via the `include_elf!("guest")` macro.

Update `host/build.rs` to compile the fibonacci guest and serialize a default input:

```rust
use std::path::PathBuf;
use zisk_sdk::{ZiskIO, ZiskStdin, build_program};

fn main() {
    // Cross-compile the guest to a RISC-V ELF
    build_program("../guest");

    // Serialize a default input (n = 10) for CLI tools
    let n = 10u32;
    let stdin = ZiskStdin::new();
    stdin.write(&n);

    let path = PathBuf::from("tmp/input.bin");
    if let Some(parent) = path.parent() {
        std::fs::create_dir_all(parent).unwrap();
    }
    stdin.save(&path).unwrap();
}
```

`build_program("../guest")` is the important line. It cross-compiles the guest and registers the ELF so `include_elf!` can find it. The rest is optional: it writes a default `input.bin` for CLI tools like `ziskemu`.

## The host

Now let's write the host that runs our fibonacci guest. Replace `host/src/main.rs` with:

```rust
use anyhow::Result;
use serde::{Deserialize, Serialize};
use zisk_sdk::{ZiskStdin, ZiskIO, ElfBinary, ProverClient, include_elf};

// Embed the compiled guest ELF at compile time
pub const ELF: ElfBinary = include_elf!("guest");

// Same struct the guest commits via io::commit()
#[derive(Serialize, Deserialize, Debug)]
struct FibResult {
    n: u32,
    value: u64,
}

fn main() -> Result<()> {
    // Prepare input for the guest
    let n = 10u32;
    let stdin = ZiskStdin::new();
    stdin.write(&n);

    // Create a client and run the guest
    let client = ProverClient::builder().emu().build().unwrap();
    let (pk, _) = client.setup(&ELF)?;
    let result = client.execute(&pk, stdin)?;

    // Read back the committed output
    println!("Cycles: {}", result.get_execution_steps());
    let output: FibResult = result.get_public_values()?;
    println!("Fibonacci({}) = {}", output.n, output.value);

    Ok(())
}
```

Here's what each piece does:

- **`include_elf!("guest")`** embeds the compiled guest ELF into the host binary at compile time. The argument matches the guest crate's package name in `guest/Cargo.toml`
- **`ZiskStdin`** is the input buffer. `.write()` serializes and appends values using bincode. It uses interior mutability, so you don't need `mut`
- **`ZiskIO`** is a trait import that `.write()` and `.save()` are defined on. Without it, those calls won't compile
- **`ProverClient::builder().emu().build()`** creates the client that runs (and later proves) your guest. `.emu()` selects the emulator backend, which works on all platforms. On Linux, use `.asm()` for significantly faster execution
- **`.setup(&ELF)`** loads the guest binary and returns a program key pair. You'll use the first value (`pk`) to run the guest. The second is for verification, covered in [Prove and Verify](/writing-programs/prove-and-verify)
- **`.execute(&pk, stdin)`** runs the guest inside the ZisK VM and returns the result, no proof involved
- **`.get_public_values::<T>()`** deserializes whatever the guest committed via `io::commit()`. The type must match what the guest wrote
- **`.get_execution_steps()`** returns how many ZisK cycles the execution took. `.get_duration()` gives wall-clock time

We'll add proving and verification to this file in [Prove and Verify](/writing-programs/prove-and-verify).

## Profiling with ziskemu

`result.get_execution_steps()` and `result.get_duration()` give you basic metrics. When you need opcode-level analysis to understand where your guest spends its proving budget, the `ziskemu` binary gives you a full cost breakdown:

```bash
cargo run --release --bin ziskemu
```

This runs the guest through a standalone emulator (separate from `ProverClient`) and reports proving cost per operation:

```
Total Cost: 12.81 sec
    Main Cost: 4.27 sec 85308 steps
    Mem Cost: 2.22 sec 222052 steps
    Opcodes: 6.24 sec 1270 steps (81182 ops)

Opcodes:
    flag: 0.00 sec (0 steps/op) (89 ops)
    copyb: 0.00 sec (0 steps/op) (10568 ops)
    add: 1.12 sec (77 steps/op) (14569 ops)
    ...
```

These numbers represent proving cost, not wall-clock time, telling you how expensive each operation is in terms of the constraint system. Use this to identify hotspots and decide whether a [precompile](/optimization/precompiles) might help.

## Other ways to run

The template includes single-stage binaries in `host/bin/` for running individual steps during development:

| Binary | Purpose |
| --- | --- |
| `execute.rs` | Run the guest without proving |
| `prove.rs` | Generate a VADCOP STARK proof |
| `compressed.rs` | Generate a compressed proof |
| `plonk.rs` | Generate a PLONK SNARK proof |
| `verify-constraints.rs` | Check PIL constraints without generating a proof |
| `ziskemu.rs` | Standalone emulator with detailed metrics |

If you've modified the guest (like we did with fibonacci), remember to update the `FibResult` type in any `bin/` files you use since the types must match what the guest commits via `io::commit()`.

The `cargo-zisk` CLI also works with pre-built ELFs directly, useful for scripting and CI. See the [CLI Reference](/cli-reference) for details.

::callout{icon="i-heroicons-information-circle"}
On macOS, add `features = ["disable_distributed"]` to both `zisk-sdk` entries in the **workspace** `Cargo.toml`. Distributed proving relies on MPI, which is not available on macOS.
::
