The primary workflow in ZisK is host-driven: you build and run the host crate, which automatically compiles the guest, feeds it input, and drives execution or proving. You do not need to manually invoke `cargo-zisk build`.

## How the build works

The host crate contains a `build.rs` script that calls `build_program("../guest")`. When you build or run any host binary, Cargo runs this build script first, which cross-compiles the guest to a RISC-V ELF targeting `riscv64ima-zisk-zkvm-elf`. The compiled ELF is then available to the host via the `include_elf!("guest")` macro.

```rust
// host/build.rs
use zisk_sdk::build_program;

fn main() {
    build_program("../guest");
}
```

You never need to run `cargo-zisk build` separately -- the guest is compiled as a dependency of the host. If the guest source hasn't changed, it won't be recompiled.

## The host binaries

The template generates several binaries under `host/bin/`, each for a different stage of the workflow:

| Binary | What it does |
| --- | --- |
| `execute` | Runs the guest, prints cycles/duration/output |
| `verify-constraints` | Checks PIL constraints without generating a proof |
| `prove` | Generates a VADCOP STARK proof and verifies it |
| `compressed` | Generates a compressed STARK proof |
| `plonk` | Generates a PLONK SNARK (for on-chain verification) |
| `ziskemu` | Standalone emulator with detailed statistics |

All of these are run from the project root using `cargo run --release --bin <name>`.

## Execute

Run the guest without generating a proof:

```bash
cargo run --release --bin execute
```

This compiles the guest (via `build.rs`), runs it in the emulator, and prints the cycle count, duration, and committed output. Behind the scenes, the execute binary creates a `ProverClient`, prepares input via `ZiskStdin`, and calls `client.execute()`:

```rust
let client = ProverClient::builder().emu().build().unwrap();
let (pk, _) = client.setup(&ELF)?;

let result = client.execute(&pk, stdin.clone())?;
println!("Cycles: {}", result.get_execution_steps());
println!("Duration: {:?}", result.get_duration());

let output: FibResult = result.get_public_values()?;
```

`.emu()` selects the emulator backend, which works on all platforms. On Linux, use `.asm()` with `.base_port(54321)` for the native x86_64 backend -- significantly faster for large programs.

The full execute binary code is covered in [Host Program](/writing-programs/host-program).

## Verify constraints

Check that the execution trace satisfies all PIL constraints before spending time on a full proof:

```bash
cargo run --release --bin verify-constraints
```

Expected output:

```
[INFO ] CstrVrfy: All global constraints were successfully verified
[INFO ] CstrVrfy: All constraints were verified
```

If a constraint fails, there's a bug in the execution trace. Better to catch it here than after a long prove run.

## Prove

Generate a VADCOP STARK proof and verify it:

```bash
cargo run --release --bin prove
```

The prove binary calls `client.prove(&pk, stdin).run()`, which generates the proof and returns a result containing the proof, public values, and verification key. Proving is covered in detail in [Prove and Verify](/proving/prove-and-verify).

## Metrics and statistics with ziskemu

The `ziskemu` binary runs the guest through a standalone emulator with detailed metrics and cost breakdowns. Unlike `execute`, which uses the `ProverClient` API, `ziskemu` uses `EmuOptions` to control what gets reported.

```bash
cargo run --release --bin ziskemu
```

The binary loads the ELF from a file path (via `elf_path!`) and reads input from the default `tmp/input.bin` that the build script generates:

```rust
use zisk_sdk::{EmuOptions, elf_path, ziskemu, ZiskStdin, ZiskIO, ElfBinaryFromFile};

let elf_path = elf_path!("guest");
let elf = ElfBinaryFromFile::new(&PathBuf::from(elf_path), false)?;
let stdin = ZiskStdin::from_file(current_dir.join("host/tmp/input.bin"))?;

let emu_options = EmuOptions { stats: true, ..EmuOptions::default() };
ziskemu(&elf, stdin, &emu_options)?;
```

With `stats: true`, the output includes a full cost breakdown -- cost definitions, total cost, register reads/writes, and per-opcode statistics:

```
Cost definitions:
    AREA_PER_SEC: 1000000 steps
    COST_MEMA_R1: 0.00002 sec
    COST_MEMA_R2: 0.00004 sec
    ...

Total Cost: 12.81 sec
    Main Cost: 4.27 sec 85308 steps
    Mem Cost: 2.22 sec 222052 steps
    Mem Align: 0.05 sec 2701 steps
    Opcodes: 6.24 sec 1270 steps (81182 ops)
    ...

Opcodes:
    flag: 0.00 sec (0 steps/op) (89 ops)
    copyb: 0.00 sec (0 steps/op) (10568 ops)
    add: 1.12 sec (77 steps/op) (14569 ops)
    ...
```

These numbers represent proving cost, not wall-clock time. They tell you how expensive each operation is in terms of the constraint system. Use this to identify where your guest spends its proving budget and whether a precompile might help.

## Execution metrics from ProverClient

The `execute` binary also provides basic metrics via the execution result:

- **`result.get_execution_steps()`** -- the number of ZisK cycles consumed
- **`result.get_duration()`** -- wall-clock execution time

For quick performance checks, these are sufficient. For detailed opcode-level analysis, use `ziskemu`.

## CLI alternative

For working with pre-built ELFs directly -- without writing a host program -- the `cargo-zisk` CLI supports the same operations:

```bash
cargo-zisk execute -e path/to/guest.elf -i input.bin -l
cargo-zisk verify-constraints -e path/to/guest.elf -i input.bin -l
cargo-zisk prove -e path/to/guest.elf -i input.bin -o /tmp/proof -l -a -y
```

The `-e` flag points to the ELF binary, `-i` provides a raw binary input file, and `-l` selects the emulator backend. This is useful for scripting, CI pipelines, or quick ad-hoc testing.

See the [CLI Reference](/cli-reference) for all flags and options.
