Now that you understand the [guest and host](/writing-programs/guest-and-host) split, let's build something. We'll write a Fibonacci sequence computation, which is the running example used throughout the rest of the docs.

## Create a project

```bash
cargo-zisk sdk new fibonacci && cd fibonacci
```

This gives you a workspace with two crates:

```
fibonacci/
├── Cargo.toml          ← workspace root
├── guest/
│   ├── Cargo.toml
│   └── src/main.rs     ← the provable computation
└── host/
    ├── Cargo.toml
    ├── build.rs        ← compiles the guest at build time
    └── src/main.rs     ← orchestrates execution and proving
```

The two-crate structure maps directly to the [guest and host](/writing-programs/guest-and-host) split. `cargo run --release` runs the host, which loads and runs the guest internally.

::project-structure-diagram
::

The template ships with a SHA-256 example. We'll replace the guest with our Fibonacci program, building it up step by step.

## Writing the guest

Open `guest/src/main.rs`. We'll replace its contents piece by piece.

### Step 1: The entry point

Every guest starts with two lines:

```rust
#![no_main]
ziskos::entrypoint!(main);

fn main() {
    // our code will go here
}
```

`#![no_main]` tells the compiler this isn't a normal binary since it runs inside ZisK's RISC-V VM, not on your OS. `ziskos::entrypoint!(main)` registers your `main` function as the VM's entry point.

---

### Step 2: Reading input

The guest receives data from the host through `io::read()`. Let's read a single number:

```rust
#![no_main]
ziskos::entrypoint!(main);

use ziskos::io;

fn main() {
    let n: u32 = io::read();
    println!("Guest received n = {}", n);
}
```

`io::read()` deserializes the next value from the input buffer. The host serializes values before execution, and the guest reads them one at a time. You can call `io::read()` multiple times and values come out in the same order the host wrote them.

---

### Step 3: The computation

Now add the actual logic. This is plain Rust. Any code that doesn't depend on OS services (filesystem, network, threads) works inside the guest:

```rust
fn main() {
    let n: u32 = io::read();
    println!("Guest received n = {}", n);

    let mut a: u64 = 0;
    let mut b: u64 = 1;
    for _ in 0..n {
        let temp = b;
        b = a + b;
        a = temp;
    }
}
```

Nothing ZisK-specific here. Just computing the nth Fibonacci number with a simple loop.

---

### Step 4: Committing output

The guest needs to publish its result. `io::commit()` writes a value to the **public values** region of the proof. Anyone who verifies the proof can see these values and confirm they came from the proven computation.

We'll commit a struct so verifiers know both what was asked (`n`) and what was computed (`value`):

```rust
use serde::{Serialize, Deserialize};

#[derive(Serialize, Deserialize, Debug)]
struct FibResult {
    n: u32,
    value: u64,
}

fn main() {
    let n: u32 = io::read();
    // ...computation...

    let result = FibResult { n, value: a };
    println!("Fibonacci({}) = {}", n, a);
    io::commit(&result);
}
```

`FibResult` derives `Serialize` and `Deserialize` because data crossing the host-guest boundary goes through [bincode](https://docs.rs/bincode/latest/bincode/) serialization. Both sides must agree on the type. We'll use the same struct definition in the host code later.

---

### The complete guest

Here's the full `guest/src/main.rs` with all the pieces together:

```rust
#![no_main]
ziskos::entrypoint!(main);

use ziskos::io;
use serde::{Serialize, Deserialize};

#[derive(Serialize, Deserialize, Debug)]
struct FibResult {
    n: u32,
    value: u64,
}

fn main() {
    let n: u32 = io::read();
    println!("Guest received n = {}", n);

    let mut a: u64 = 0;
    let mut b: u64 = 1;
    for _ in 0..n {
        let temp = b;
        b = a + b;
        a = temp;
    }

    let result = FibResult { n, value: a };
    println!("Fibonacci({}) = {}", n, a);
    io::commit(&result);
}
```

Replace the template's SHA-256 code with this.

## Understanding the dependencies

The template generates all `Cargo.toml` files for you. The generated files include dependencies for the SHA-256 example (`sha2`, `byteorder`). Since our fibonacci guest doesn't need them, the listings below show the cleaned-up versions.

### Workspace root: `Cargo.toml`

Declares the shared ZisK dependencies that both crates reference:

```toml
[workspace]
members = ["host", "guest"]
resolver = "2"

[workspace.dependencies]
zisk-sdk = { git = "https://github.com/0xPolygonHermez/zisk.git", branch = "pre-develop-0.16.0" }
ziskos = { git = "https://github.com/0xPolygonHermez/zisk.git", branch = "pre-develop-0.16.0" }
```

- **`ziskos`** is the guest-side SDK (`entrypoint!`, `io::read()`, `io::commit()`)
- **`zisk-sdk`** is the host-side SDK (`ProverClient`, `ZiskStdin`, build tooling)

You can remove `sha2` and `byteorder` from `[workspace.dependencies]` if they're still listed from the template.

---

### Guest: `guest/Cargo.toml`

```toml
[package]
name = "guest"
version = "0.1.0"
edition = "2024"

[dependencies]
ziskos = { workspace = true }
serde = { version = "1.0", default-features = false, features = ["derive"] }
```

`serde` uses `default-features = false` to avoid pulling in features the guest doesn't need, and the `derive` feature gives us `#[derive(Serialize, Deserialize)]`. Remove `sha2` and `byteorder` if still present from the template.

---

::callout{icon="i-heroicons-information-circle"}
The only file you must change is `guest/src/main.rs`. Removing the unused SHA-256 dependencies from the Cargo.toml files is optional but keeps things clean. If your guest needs additional crates, add them to `guest/Cargo.toml`.
::
