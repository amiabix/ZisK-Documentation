You've seen a ZisK program take input, compute a result, and produce a proof. But that proof only works because ZisK can guarantee: *the same program with the same input will always produce the same output.* To make that guarantee, ZisK splits every project into two programs:

- **Guest**: the computation that runs inside the ZisK VM and gets proven. No filesystem, no network, no randomness — pure deterministic logic.
- **Host**: a normal Rust program that runs on your machine. It gathers input, launches the guest, and handles the proof.

## Guest program

The guest is the code you want to prove. It is a Rust program that compiles to RISC-V and executes inside the ZisK virtual machine.

The VM environment is fully deterministic: no filesystem, no network, no threads, no system clock, no randomness. Given the same input, the guest always produces the same output. This determinism is what makes the execution provable.

Here is a guest that reads a number, hashes some data that many times, and commits the result:

```rust
#![no_main]
ziskos::entrypoint!(main);

use ziskos::io;
use sha2::{Sha256, Digest};

fn main() {
    let n: u32 = io::read();
    let mut hash = Sha256::digest(b"zisk");

    for _ in 1..n {
        hash = Sha256::digest(&hash);
    }

    io::commit(&hash);
}
```

Two functions handle all communication with the outside world:

- `io::read()` reads input that the host prepared before execution.
- `io::commit()` writes output that becomes part of the provable result.

The logic between those two calls is standard Rust. Any crate that does not depend on OS services works inside the guest.

`#![no_main]` tells the compiler this is not a standalone binary. `ziskos::entrypoint!(main)` registers the function as the VM's entry point.

## Why the split

A proof is a statement: *this program, given this input, produced this output.* For that to be verifiable, execution must be reproducible. If the guest could access a network API, two executions could produce different results, and the verifier would have no way to confirm the output.

The sandbox enforces this: the same program with the same input always produces the same result. But a sandboxed program cannot read files, call APIs, or decide how to handle its own output. It needs a partner to do all of that.

That partner is the host.

---

## Host program

The host runs on your machine with no restrictions. It has full access to the filesystem, network, and everything else.

Its job is to set up the guest and handle results:

1. Prepare input data and pass it to the VM.
2. Run the guest (execute for testing, or prove for a real proof).
3. Read back the output and do something with it.

You will write the host in the [Build and Run](/writing-programs/build-and-run) section. For now, what matters is the relationship: the host decides *what* to run and *with what data*. The guest does the computation that gets proven.

---

## How they communicate

::guest-i-o-diagram
::

Data flows in one direction at a time. The host writes all input before the guest starts. The guest reads that input, computes, and commits its output. After execution finishes, the host reads the output. There is no communication during execution.

The two channels have different visibility:

**Input** is **private**. The host writes data into a stdin buffer, and the guest reads it with `io::read()`. This data is not included in the proof unless the guest explicitly commits it.

**Output** is **public**. The guest writes results with `io::commit()`, producing **public values** that are cryptographically bound to the proof. These are the only part of the execution that a verifier can inspect.

---

## What goes where

A useful rule of thumb as you start building: if a verifier needs to trust that a computation happened correctly, it belongs in the guest. Everything else belongs in the host.

You will see this principle in action throughout the following sections as you write your first guest, build the host, and generate proofs.
