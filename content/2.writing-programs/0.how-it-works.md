The [Quickstart](/getting-started/getting-started-with-zisk) ran everything with a single `cargo run --release`. That's convenient, but it hides what's actually happening. This page breaks the process into individual steps so you can see how a ZisK program goes from source code to a verified proof.

::pipeline-diagram
::

We'll use the same `sha_hasher` project from the quickstart. If you haven't created it yet:

```bash
cargo-zisk sdk new sha_hasher
cd sha_hasher
```

---

## The project

The template has a [guest and host](/writing-programs/guest-and-host) like every ZisK project. The guest (`guest/src/main.rs`) reads a number `n`, hashes some data with SHA-256 `n` times, and outputs the final hash. `build.rs` generates an `input.bin` with `n = 1000` so you have something to run right away.

---

## Build

The first step is compiling the guest into a format the ZisK VM can execute:

```bash
cargo-zisk build --release
```

This cross-compiles the guest to a **RISC-V ELF binary**, the instruction set that ZisK's virtual machine understands. The resulting file lands at:

```
target/riscv64ima-zisk-zkvm-elf/release/sha_hasher
```

This is a standard ELF binary, just compiled for RISC-V instead of your machine's native architecture. The VM will interpret these instructions and record every step during execution.

---

## Execute

Before generating a proof (which takes time), you can run the program through ZisK's emulator to verify it works correctly:

```bash
ziskemu -e target/riscv64ima-zisk-zkvm-elf/release/sha_hasher -i build/input.bin
```

- `-e` points to the ELF binary you just built
- `-i` points to the serialized input file that `build.rs` generated

The output shows the SHA-256 hash as 32-byte public values, split into eight 4-byte words:

```
98211882
bd13089b
6ccf1fca
81f7f0e4
abf6352a
0c39c9b1
1f142cac
233f1280
```

These are the **public values**, the guest's committed output. When you later generate a proof, these values are cryptographically bound to it. Anyone who verifies the proof can see them and confirm they came from the proven computation, without re-running the program.

You can also build and execute in one step:

```bash
cargo-zisk run --release -i build/input.bin
```

---

## Prove

Execution confirms the program works, but it doesn't produce a proof. To generate one, there are two steps.

First, generate the **program setup** files. These are derived from the program's structure and are reusable across different inputs:

```bash
cargo-zisk rom-setup -e target/riscv64ima-zisk-zkvm-elf/release/sha_hasher
```

Then generate and verify the proof:

```bash
cargo-zisk prove -e target/riscv64ima-zisk-zkvm-elf/release/sha_hasher -i build/input.bin -o proof -a -y
```

- `-e` is the ELF binary
- `-i` is the input
- `-o proof` saves the proof to the `./proof` directory
- `-a` aggregates sub-proofs into a single final proof
- `-y` automatically verifies the proof after generation

When proving finishes, you'll see:

```
[INFO ] ProofMan:     ✓ Vadcop Final proof was verified
[INFO ] ProofMan: Proofs generated successfully
```

What happened here is the core of what ZisK does. The prover replayed the guest's execution, recorded every instruction as an **execution trace**, split that trace across specialized **state machines** (one for arithmetic, one for memory, one for binary operations, etc.), proved each independently, and tied them together into a single VADCOP STARK proof.

The result is a compact proof that says: "this program, given this input, produced this output." Anyone can verify that statement without re-running the computation.

---

## Verify

To verify an existing proof independently:

```bash
cargo-zisk verify -p ./proof/vadcop_final_proof.bin
```

Verification is fast. It checks the mathematical structure of the proof without re-executing the program. This is the whole point: proving is expensive, but verification is cheap.
