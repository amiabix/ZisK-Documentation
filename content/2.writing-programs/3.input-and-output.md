In [Build and Run](/writing-programs/build-and-run) you saw the host send `n = 10` to the guest and read back `Fibonacci(10) = 55`. This page explains how that data flow works under the hood.

## How data flows

Data flows in one direction at a time. The host writes all input before execution begins, the guest reads that input during execution and commits its output, and after execution the host reads the committed output. There is no interactive communication: no back-and-forth, no callbacks, no shared memory.

![Data flow between host and guest](/images/data-flow.svg)

</div>

Think of it as two sequential tapes:

```
Input tape:   [  n  |  data  |  config  ]
                 ^
              read head (moves right with each io::read())

Output tape:  [  hash  |  count  ]
                          ^
              write head (moves right with each io::commit())
```

Read/write forward only, no random access, no seeking. The order of `write()` and `read()` calls is a contract between host and guest: reorder one side, reorder the other to match.

This one-way model ensures determinism. If the guest could call back to the host, different hosts could provide different responses, breaking reproducibility.

## Sending input

On the host side, you build up an input buffer using `ZiskStdin`:

```rust
let stdin = ZiskStdin::new();
stdin.write(&10u32);
stdin.write(&"hello".to_string());
stdin.write(&vec![1u8, 2, 3]);
```

Each call to `.write(&value)` serializes the value using [bincode](https://github.com/bincode-org/bincode) and appends the bytes to the buffer. You can call `.write()` as many times as you like and values are appended in order.

For raw bytes that are already serialized or don't need serialization, use `.write_slice()`:

```rust
let raw_data: &[u8] = &[0xDE, 0xAD, 0xBE, 0xEF];
stdin.write_slice(raw_data);
```

`ZiskStdin` also has constructors for loading input from a file or wrapping an existing byte vector:

```rust
// From a file on disk (e.g., a pre-generated input.bin)
let stdin = ZiskStdin::from_file("tmp/input.bin")?;

// From a raw byte vector
let stdin = ZiskStdin::from_vec(vec![1, 2, 3, 4]);
```

You can save a buffer to disk with `stdin.save(&path)`, which is what `build.rs` does to create the default `input.bin`.

## Reading input

On the guest side, each call to `io::read::<T>()` deserializes the next value from the buffer:

```rust
let n: u32 = io::read();           // reads the 10u32
let msg: String = io::read();       // reads "hello"
let bytes: Vec<u8> = io::read();    // reads [1, 2, 3]
```

The two sides must agree on types and order. If the host writes a `u32` followed by a `String`, the guest must read a `u32` followed by a `String` in that order. If the types don't match, deserialization fails and the guest panics. Reading past the end of the input also panics: these are hard failures, not recoverable errors.

For raw bytes (written with `.write_slice()` on the host), use `io::read_vec()`:

```rust
let raw: Vec<u8> = io::read_vec();  // returns all remaining input bytes
```

Unlike `io::read()` which deserializes one value at a time, `read_vec()` returns the entire remaining buffer as raw bytes.

A common pattern is to write a single struct that contains all the input the guest needs, which avoids ordering mistakes:

```rust
// Shared type
#[derive(Serialize, Deserialize)]
struct ProgramInput {
    iterations: u32,
    seed: [u8; 32],
    data: Vec<u8>,
}

// Host
stdin.write(&input);

// Guest
let input: ProgramInput = io::read();
```

Instead of writing and reading five separate values in the right sequence, you write and read one struct.

## Committing output

`io::commit(&value)` serializes the value with bincode and writes it to the **public output region**. This data becomes part of the proof: verifiers can read it and confirm it came from the proven computation.

```rust
// Inside the guest
let result = FibResult { n: 10, value: 55 };
io::commit(&result);
```

You can call `io::commit()` multiple times and each call appends to the same output region.

::callout{icon="i-heroicons-exclamation-triangle"}
The public output region is **256 bytes** (64 `u32` slots). Exceeding this fails proving. For large outputs, commit a hash and provide the full data alongside the proof.
::

The guest also has `io::write(buf)` which writes raw bytes directly to the same output region in 4-byte chunks, converting each to a little-endian `u32`. `io::commit()` uses `io::write()` internally after serializing. Use `io::commit()` for structured data and `io::write()` only when you need byte-level control.

## Reading output

After execution, the host reads the committed output with `.get_public_values::<T>()`:

```rust
let result = client.execute(&pk, stdin)?;

let output: FibResult = result.get_public_values()?;
println!("fib({}) = {}", output.n, output.value);
```

The type must match what the guest committed via `io::commit()`. This method is available on all result types: execution results, prove results, and verify-constraints results.

You can also access the raw bytes through `result.get_publics()`, which returns a `ZiskPublics` struct. This is useful when you need raw bytes for on-chain verification or custom processing.

## Serialization

ZisK uses bincode for all serialization between host and guest. Bincode is a compact binary format with no field names or delimiters, which keeps the input buffer small since the entire input becomes part of the execution trace.

| Type | Bincode size |
| --- | --- |
| `u8` / `i8` / `bool` | 1 byte |
| `u32` / `i32` | 4 bytes |
| `u64` / `i64` | 8 bytes |
| `u128` / `i128` | 16 bytes |
| `String` / `Vec<T>` | 8 bytes (length as `u64`) + data |

Integers are encoded in little-endian byte order. Strings and vectors are length-prefixed with a `u64` length followed by the raw contents. Structs are serialized field by field in declaration order with no field names in the binary representation.

Both guest and host must agree on the struct definition. For larger projects, put shared types in a common crate that both depend on.

## Quick reference

### Host side (`ZiskStdin`)

| Method | Description |
| --- | --- |
| `ZiskStdin::new()` | Creates an empty input buffer. Call `.write()` to add values |
| `ZiskStdin::from_file(path)` | Loads a previously saved input buffer from disk |
| `ZiskStdin::from_vec(data)` | Wraps an existing `Vec<u8>` as the input buffer |
| `.write(&value)` | Serializes with bincode and appends to the buffer. Value must implement `Serialize` |
| `.write_slice(&[u8])` | Appends raw bytes without serialization |
| `.save(&path)` | Writes the buffer to a file on disk |

### Guest side (`ziskos::io`)

| Function | Description |
| --- | --- |
| `io::read::<T>()` | Deserializes the next value of type `T` from the input buffer. Panics if deserialization fails or the buffer is exhausted |
| `io::read_vec()` | Returns all remaining input bytes as a `Vec<u8>` without deserializing |
| `io::commit(&value)` | Serializes with bincode and writes to the public output region |
| `io::write(buf)` | Writes raw bytes directly to the public output region in 4-byte chunks |

### Result reading (host side)

| Method | Description |
| --- | --- |
| `result.get_public_values::<T>()` | Deserializes committed public output into a typed value |
| `result.get_publics()` | Returns the raw `ZiskPublics` struct for on-chain verification or custom processing |
