ZisK supports three proof types, each with different trade-offs between size, generation time, and where you can verify them.

::proof-types-diagram
::

| Type | Size | On-chain verifiable | How to generate |
| --- | --- | --- | --- |
| VADCOP STARK | Large | No | `.run()?` |
| Compressed | Medium | No | `.compressed().run()?` |
| PLONK SNARK | Small (constant) | Yes | `.plonk().run()?` |

All three are cryptographically sound. The difference is practical: VADCOP proofs are fast to generate but too large for a blockchain transaction. PLONK proofs are compact enough to verify in a smart contract. Compressed proofs sit in between.

All examples on this page use `.emu()` for simplicity. For production, use `.asm()`. It runs the native ZisK assembly pipeline and is significantly faster.

---

## VADCOP STARK

VADCOP (Virtually Aggregated Distributed Computation Over Polynomials) is ZisK's native proof system. The prover replays the guest's execution, records every instruction, and splits the resulting trace across 35 specialized state machines (AIRs) — arithmetic, memory, binary operations, cryptographic precompiles (Keccak, SHA-256, Poseidon2, Blake2), and others. Each state machine is proven independently as a Goldilocks FRI STARK, then aggregated through a recursive tree into a single proof.

::proving-pipeline-diagram
::

This is the default. When you call `.run()` without selecting a proof type, you get a VADCOP STARK:

```rust
let result = client.prove(&pk, stdin)
    .with_proof_options(proof_opts)
    .run()?;

client.verify(result.get_proof(), result.get_publics(), result.get_program_vk())?;
```

VADCOP proofs are the fastest to generate, making them ideal for development, testing, and any scenario where you only need off-chain verification.

## How the types relate

All three proof types start from the same VADCOP Final proof. Compressed and PLONK are independent transformations of that base — you choose one path or the other, not both in sequence.

::proof-branching-diagram
::

::callout{icon="i-heroicons-exclamation-triangle"}
You cannot convert a compressed proof to PLONK. Both two-step APIs (`compress()` and `prove_snark()`) only accept an uncompressed VADCOP Final proof. If you need PLONK, start from the uncompressed VADCOP.
::

## Compressed

A compressed proof is a recursive STARK that *verifies* the VADCOP Final proof. A Circom circuit encodes the VADCOP verification logic, and the prover generates a new STARK over that circuit with parameters optimized for smaller proof size (different trace size, blowup factor, and query count via `starkinfo.json`). The result is still a Goldilocks FRI proof, but significantly smaller — the verification circuit is simpler than the original 35 state machines.

::compression-pipeline-diagram
::

**One step** — ProofMan generates the compressed proof inline:

```rust
let result = client.prove(&pk, stdin)
    .with_proof_options(proof_opts)
    .compressed()
    .run()?;

client.verify(result.get_proof(), result.get_publics(), result.get_program_vk())?;
```

**Two steps** — generate a VADCOP proof first, then compress it:

```rust
// Step 1: Generate the VADCOP proof
let vadcop_result = client.prove(&pk, stdin)
    .with_proof_options(proof_opts)
    .run()?;

// Step 2: Compress it
let compressed_result = client.compress(
    vadcop_result.get_proof(),
    vadcop_result.get_publics(),
    &vk,
)?;
```

Use compressed proofs when you need something smaller than VADCOP for storage or transmission, or when you don't need on-chain verification.

## PLONK SNARK

PLONK produces a constant-size proof (768 bytes) that can be verified on-chain. It is the only proof type the Solidity verifier accepts. Getting there requires a three-stage transformation from the Goldilocks-field VADCOP proof to a BN128-field SNARK.

::snark-pipeline-diagram
::

**Stage 1 — RecursiveF (field transition):** A Circom circuit verifies the Goldilocks STARK using BN128 arithmetic. This is the bridge between ZisK's native field (Goldilocks, 64-bit) and the EVM-compatible field (BN128/BN254, 254-bit). The output is a STARK proof over BN128.

**Stage 2 — Witness generation:** A compiled Circom circuit (`final.dylib`) verifies the BN128 STARK and produces a witness vector for the SNARK prover.

**Stage 3 — SNARK proof:** The PLONK (or Fflonk) prover, initialized from a `final.zkey` file, generates the final proof: 24 `uint256` values (768 bytes) plus a public input digest. This is what the Solidity `PlonkVerifier.verifyProof(uint256[24], uint256[1])` checks.

There's one extra requirement: you must call `.snark()` when building the `ProverClient`. This loads the SNARK proving key and initializes the circuit infrastructure:

```rust
let client = ProverClient::builder()
    .emu()
    .snark()
    .build()
    .unwrap();

let (pk, vk) = client.setup(&ELF)?;

let result = client.prove(&pk, stdin)
    .plonk()
    .run()?;

client.verify(result.get_proof(), result.get_publics(), result.get_program_vk())?;
```

::callout{icon="i-heroicons-exclamation-triangle"}
`.snark()` must be called on the builder before `.build()`. If you skip it and try to generate a PLONK proof, the proving pipeline will fail at runtime. This is the most common mistake when working with PLONK proofs.
::

**Two steps** — generate a VADCOP proof first, then convert it:

```rust
let vadcop_result = client.prove(&pk, stdin).run()?;

let snark_result = client.prove_snark(
    vadcop_result.get_proof(),
    vadcop_result.get_publics(),
    &vk,
)?;
```

Use PLONK when you need on-chain verification, when proof size must be minimal, or for production deployments that submit proofs to smart contracts. See [On-Chain Verification](/writing-programs/onchain-verification) for deployment details.

## Choosing a Proof Type

- **Development and testing**: VADCOP STARK. Fastest generation, no extra setup.
- **On-chain verification**: PLONK SNARK. Only type the Solidity verifier accepts.
- **Smaller off-chain proofs**: Compressed. Smaller than VADCOP, no SNARK setup overhead.

Most projects use VADCOP during development and switch to PLONK for production.

For details on `ProofOpts`, saving and loading proofs, and the full `ProveBuilder` API, see [Prove and Verify](/writing-programs/prove-and-verify).
