In the [Quickstart](/getting-started/getting-started-with-zisk) you wrote a Fibonacci guest, built it, and proved it. Here's what each piece of that code does.

## Project structure

Every ZisK project is a Rust workspace with two crates: the **guest** (runs inside the zkVM) and the **host** (runs on your machine and drives everything). Here's the layout that `cargo-zisk sdk new` created:

```
fibonacci/
├── Cargo.toml
├── guest/
│   ├── Cargo.toml
│   └── src/main.rs
└── host/
    ├── Cargo.toml
    ├── build.rs
    ├── src/main.rs
    └── bin/
        ├── execute.rs
        ├── prove.rs
        ├── compressed.rs
        ├── plonk.rs
        ├── verify-constraints.rs
        └── ziskemu.rs
```

The guest crate targets ZisK's RISC-V VM. The host crate targets your native machine and uses the ZisK SDK to compile the guest, feed it input, and trigger execution or proving. The host `bin/` directory contains separate binaries for different workflows.

## The guest program

```rust
#![no_main]
ziskos::entrypoint!(main);

use ziskos::io;
use serde::{Serialize, Deserialize};

#[derive(Serialize, Deserialize, Debug)]
struct FibResult {
    n: u32,
    value: u64,
}

fn main() {
    let n: u32 = io::read();
    println!("Guest received n = {}", n);

    let mut a: u64 = 0;
    let mut b: u64 = 1;
    for _ in 0..n {
        let temp = b;
        b = a + b;
        a = temp;
    }

    let result = FibResult { n, value: a };
    println!("Fibonacci({}) = {}", n, a);
    io::commit(&result);
}
```

### `#![no_main]`

The guest runs inside ZisK's RISC-V VM, not as a normal binary. `#![no_main]` tells the compiler not to emit a standard entry point.

### `ziskos::entrypoint!(main)`

Registers `main` as the guest's entry point. The VM calls this function on startup.

### `io::read()`

Reads the next value from the host's input buffer. The host serializes with [bincode](https://docs.rs/bincode/latest/bincode/), the guest deserializes. You can call `io::read()` multiple times; values are consumed in the order the host wrote them.

### `io::commit()`

Writes data to the **public values** region of the proof. Committed values are visible to anyone who verifies the proof. Here we commit both `n` and the result so verifiers know what was computed.

### `println!`

Works during emulation (prints to terminal). Silently discarded during proving.

### Serialization with `serde`

`FibResult` derives `Serialize` and `Deserialize` so both guest and host can work with the same structured type across the bincode boundary.

## The guest dependencies

The workspace `Cargo.toml` at the project root declares shared dependencies:

```toml
[workspace]
members = ["host", "guest"]
resolver = "2"

[workspace.dependencies]
zisk-sdk = { git = "https://github.com/0xPolygonHermez/zisk.git", branch = "0.16" }
ziskos = { git = "https://github.com/0xPolygonHermez/zisk.git", branch = "0.16" }
```

The `guest/Cargo.toml` references these via `workspace = true`:

```toml
[package]
name = "guest"
version = "0.1.0"
edition = "2024"

[dependencies]
ziskos = { workspace = true }
serde = { version = "1.0", default-features = false, features = ["derive"] }
```

`ziskos` provides the guest-side API (`entrypoint!`, `io::read()`, `io::commit()`). `serde` needs `default-features = false` because the guest runs in a constrained VM environment -- the full `std` serde feature set is not needed. The `derive` feature enables `#[derive(Serialize, Deserialize)]`.

## What the guest can and cannot do

The guest is ordinary Rust. You have access to `std` (`Vec`, `HashMap`, `String`, etc.) and third-party crates that compile to RISC-V. Most do.

No filesystem, no network, no threads, no clock. The guest is a pure computation: same input always produces the same output.
