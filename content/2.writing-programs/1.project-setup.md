The [Quickstart](/getting-started/getting-started-with-zisk) ran the default SHA-256 template to confirm your install works. Now we'll write our own program: a Fibonacci sequence computation, which is the running example used throughout the rest of the docs.

## Create a project

```bash
cargo-zisk sdk new fibonacci && cd fibonacci
```

This gives you a workspace with two crates:

```
fibonacci/
├── Cargo.toml
├── guest/
│   ├── Cargo.toml
│   └── src/main.rs
└── host/
    ├── Cargo.toml
    ├── build.rs
    ├── src/main.rs
    └── bin/
        ├── execute.rs
        ├── prove.rs
        ├── compressed.rs
        ├── plonk.rs

        └── ziskemu.rs
```

The **guest** is the program that runs inside the ZisK virtual machine, the computation you want to prove. The **host** runs on your machine and orchestrates everything: compiling the guest, feeding it input, and driving execution or proving. `host/src/main.rs` is the primary host program that `cargo run --release` executes, while the `host/bin/` directory has optional single-stage binaries useful during development.

The template ships with a SHA-256 example. We'll replace the guest with our Fibonacci program.

## Write the guest

Replace `guest/src/main.rs` with:

```rust
#![no_main]
ziskos::entrypoint!(main);

use ziskos::io;
use serde::{Serialize, Deserialize};

#[derive(Serialize, Deserialize, Debug)]
struct FibResult {
    n: u32,
    value: u64,
}

fn main() {
    let n: u32 = io::read();
    println!("Guest received n = {}", n);

    let mut a: u64 = 0;
    let mut b: u64 = 1;
    for _ in 0..n {
        let temp = b;
        b = a + b;
        a = temp;
    }

    let result = FibResult { n, value: a };
    println!("Fibonacci({}) = {}", n, a);
    io::commit(&result);
}
```

Let's walk through what's happening here.

### VM entry point

```rust
#![no_main]
ziskos::entrypoint!(main);
```

Every guest needs these two lines. `#![no_main]` tells the compiler this isn't a normal binary since it runs inside ZisK's RISC-V VM, not on your OS. `ziskos::entrypoint!(main)` registers your `main` function as the VM's entry point.

### Reading input

```rust
let n: u32 = io::read();
```

`io::read()` is how the guest receives input from the host. The host serializes values before execution, and the guest deserializes them one at a time. Here we read a single `u32` representing the Fibonacci index, but you can call `io::read()` multiple times and values come out in the same order the host wrote them.

### The computation

```rust
let mut a: u64 = 0;
let mut b: u64 = 1;
for _ in 0..n {
    let temp = b;
    b = a + b;
    a = temp;
}
```

Plain Rust, nothing special. Just computing the nth Fibonacci number with a simple loop. Any valid Rust code works inside the guest.

### Committing output

```rust
io::commit(&result);
```

`io::commit()` writes the result to the **public values** region of the proof. This is the guest's output: anyone who verifies the proof can see these values and confirm they came from the proven computation. We commit a `FibResult` struct so verifiers know both what was asked (`n`) and what was computed (`value`).

### Shared types

```rust
#[derive(Serialize, Deserialize, Debug)]
struct FibResult {
    n: u32,
    value: u64,
}
```

`FibResult` derives `Serialize` and `Deserialize` because data crossing the host-guest boundary goes through [bincode](https://docs.rs/bincode/latest/bincode/) serialization, so both sides need to agree on the type. We'll use the same struct definition in the host code later.

## Understanding the dependencies

The template generates all `Cargo.toml` files for you, so you don't need to change them. Here's what's in each one and how the pieces connect.

---

### Workspace root: `Cargo.toml`

Declares the shared ZisK dependencies that both crates reference:

```toml
[workspace]
members = ["host", "guest"]
resolver = "2"

[workspace.dependencies]
zisk-sdk = { git = "https://github.com/0xPolygonHermez/zisk.git", branch = "0.16" }
ziskos = { git = "https://github.com/0xPolygonHermez/zisk.git", branch = "0.16" }
```

- **`ziskos`** is the guest-side SDK (`entrypoint!`, `io::read()`, `io::commit()`)
- **`zisk-sdk`** is the host-side SDK (`ProverClient`, `ZiskStdin`, build tooling)

---

### Guest: `guest/Cargo.toml`

```toml
[package]
name = "guest"
version = "0.1.0"
edition = "2024"

[dependencies]
ziskos = { workspace = true }
serde = { version = "1.0", default-features = false, features = ["derive"] }
```

`serde` needs `default-features = false` because the guest runs in a constrained environment, and the `derive` feature gives us `#[derive(Serialize, Deserialize)]`.

---

### Host: `host/Cargo.toml`

```toml
[package]
name = "host"
version = "0.1.0"
edition = "2024"
default-run = "host"

[dependencies]
zisk-sdk = { workspace = true }
anyhow = "1.0"
serde = { version = "1.0", default-features = false, features = ["derive"] }

[build-dependencies]
zisk-sdk = { workspace = true }
```

- **`zisk-sdk`** appears twice: under `[dependencies]` for the host code, and under `[build-dependencies]` for the `build.rs` script that compiles the guest
- **`default-run = "host"`** tells Cargo that `cargo run` should run `src/main.rs` by default

---

::callout{icon="i-heroicons-information-circle"}
The only file you need to change from the template is `guest/src/main.rs`. Everything else is ready to go. If your guest needs additional crates (like `sha2` or `hex`), add them to `guest/Cargo.toml`.
::
