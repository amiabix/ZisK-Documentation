ZisK supports three proof types, each trading off size, generation time, and capabilities.

## Overview

| Type | Size | On-chain verifiable | How to generate |
| --- | --- | --- | --- |
| VADCOP STARK | Large | No | `client.prove(&pk, stdin).run()?` |
| Compressed | Medium | No | `.compressed().run()?` |
| PLONK SNARK | Small (constant size) | Yes | `.plonk().run()?` (requires `.snark()` at build) |

All three proof types are sound. The difference is practical: VADCOP proofs are fast to generate but too large for a blockchain transaction. PLONK proofs are compact enough to verify in a smart contract. Compressed proofs sit in between.

## VADCOP STARK

VADCOP (Virtually Aggregated Distributed Computation Over Polynomials) is ZisK's native proof system. It splits the trace across specialized state machines, proves each independently, and ties them together with a virtual bus.

Default proof type:

```rust
use anyhow::Result;
use zisk_sdk::{include_elf, ElfBinary, ProverClient, ZiskIO, ZiskStdin};

const ELF: ElfBinary = include_elf!("my-guest-crate");

fn main() -> Result<()> {
    let client = ProverClient::builder().emu().build().unwrap();
    let (pk, vk) = client.setup(&ELF)?;

    let stdin = ZiskStdin::new();
    stdin.write(&42u64);

    // Default proof type is VADCOP STARK
    let result = client.prove(&pk, stdin).run()?;

    // Verify it
    client.verify(result.get_proof(), result.get_publics(), &vk)?;
    println!("VADCOP proof verified");

    Ok(())
}
```

**When to use VADCOP STARK:**
- During development and testing
- When you only need off-chain verification
- When you want the fastest possible proof generation

## Compressed Proof

Wraps the VADCOP STARK in a recursive proof: the prover proves "I verified a VADCOP proof," producing a smaller proof for the same computation.

```rust
let client = ProverClient::builder().emu().build().unwrap();
let (pk, vk) = client.setup(&ELF)?;

let stdin = ZiskStdin::new();
stdin.write(&42u64);

let result = client.prove(&pk, stdin)
    .compressed()
    .run()?;

client.verify(result.get_proof(), result.get_publics(), &vk)?;
println!("Compressed proof verified");
```

Chain `.compressed()` then `.run()`.

**When to use compressed:**
- When you need a smaller proof for storage or transmission
- When you are aggregating multiple proofs together
- When you want something smaller than VADCOP but do not need on-chain verification

## PLONK SNARK

Small constant-size proof, verifiable on-chain. Requires `.snark()` on the builder:

```rust
// Note: .snark() is called before .build()
let client = ProverClient::builder()
    .emu()
    .snark()
    .build()
    .unwrap();

let (pk, vk) = client.setup(&ELF)?;

let stdin = ZiskStdin::new();
stdin.write(&42u64);

let result = client.prove(&pk, stdin)
    .plonk()
    .run()?;

client.verify(result.get_proof(), result.get_publics(), &vk)?;
println!("PLONK proof verified");
```

`.snark()` initializes the SNARK circuit once at build time, reused for all subsequent PLONK proofs.

::callout{icon="i-heroicons-exclamation-triangle"}
`.snark()` must be called on the builder before `.build()`. If you forget it and then try to generate a PLONK proof, the proving pipeline will fail at runtime. This is the single most common mistake when working with PLONK proofs.
::

**When to use PLONK SNARK:**
- When you need on-chain verification
- When proof size must be minimal
- For production deployments that submit proofs to smart contracts

See [On-Chain Verification](/proving-in-depth/onchain-verification) for deployment details.

## The ProveBuilder API

`client.prove(&pk, stdin)` returns a `ProveBuilder`. Configure proof type, then `.run()`:

```rust
// VADCOP STARK (default)
let result = client.prove(&pk, stdin).run()?;

// Compressed
let result = client.prove(&pk, stdin)
    .compressed()
    .run()?;

// PLONK SNARK
let result = client.prove(&pk, stdin)
    .plonk()
    .run()?;

// With custom options
let result = client.prove(&pk, stdin)
    .with_proof_options(opts)
    .plonk()
    .run()?;
```

The `ProveBuilder` methods are:

- **`.compressed()`**: select compressed proof output
- **`.plonk()`**: select PLONK SNARK proof output
- **`.with_proof_options(opts)`**: pass custom `ProofOpts` (see below)
- **`.run()`**: execute the proving pipeline and return the result

`.run()` is required. The builder is lazy.

## ProofOpts

Custom proving options:

```rust
use zisk_sdk::ProofOpts;

let opts = ProofOpts::default().minimal_memory();

let result = client.prove(&pk, stdin)
    .with_proof_options(opts)
    .run()?;
```

Methods: `.minimal_memory()`, `.save_proofs()`, `.verify_proofs()`, `.no_aggregation()`, `.output_dir(path)`. Defaults work for most cases.

## Saving and Loading Proofs

```rust
// Save a proof with its public values and verification key
result.save_proof_with_publics("/tmp/my-proof")?;

// Load it back
use zisk_sdk::ZiskProofWithPublicValues;
let loaded = ZiskProofWithPublicValues::load("/tmp/my-proof")?;

// Verify the loaded proof
client.verify(&loaded.proof, &loaded.publics, &loaded.program_vk)?;
```

Or save just the proof:

```rust
// Save just the proof
result.get_proof().save("/tmp/my-proof-only")?;

// Load it back
use zisk_sdk::ZiskProof;
let proof = ZiskProof::load("/tmp/my-proof-only")?;
```

## Choosing a Proof Type

- **Development/testing**: VADCOP STARK (default, fastest)
- **On-chain verification**: PLONK SNARK (only type the Solidity verifier accepts)
- **Smaller off-chain proofs**: Compressed (no SNARK setup overhead)
- **Aggregation pipelines**: Start with compressed, then aggregate

Most projects use VADCOP in development and PLONK in production.
