`ProverClient` is the central SDK type. It handles execution, proving, and verification. This page is the complete reference for its builder API and configuration.

## The Builder Pattern

ProverClient uses a typestate builder. Each method returns a new type, so the compiler enforces correct ordering. You cannot call `.build()` without first choosing a backend.

```rust
use zisk_sdk::ProverClient;

let prover = ProverClient::builder()
    .emu()          // step 1: choose a backend
    .prove()        // step 2 (optional): choose a mode
    .build()        // step 3: finalize, returns Result
    .unwrap();
```

Minimal builder:

```rust
let prover = ProverClient::builder()
    .emu()
    .build()
    .unwrap();
```

Emulator backend, default prove mode. Sufficient for most development workflows.

## Backends

The backend determines how your guest ELF is executed. ZisK offers two backends.

### Emulator Backend

```rust
let prover = ProverClient::builder()
    .emu()
    .build()
    .unwrap();
```

Interprets RISC-V instructions in software. Works on all platforms (macOS, Linux).

### ASM Backend

```rust
let prover = ProverClient::builder()
    .asm()
    .build()
    .unwrap();
```

Compiles the guest to native x86_64 assembly. Faster than emulator, Linux x86_64 only.

Custom ASM options:

```rust
let prover = ProverClient::builder()
    .asm()
    .base_port(54321)          // custom base port
    .unlock_mapped_memory(true) // unlock mapped memory
    .no_auto_setup(true)        // disable automatic setup
    .build()
    .unwrap();
```

Start with `.emu()`. Switch to `.asm()` for Linux CI or production.

## Modes

Modes determine what happens with the execution trace after running the guest.

### Prove Mode

```rust
let prover = ProverClient::builder()
    .emu()
    .prove()
    .build()
    .unwrap();
```

Default mode. Generates a full proof. Omitting `.prove()` gives the same result.

### Verify Constraints Mode

```rust
let prover = ProverClient::builder()
    .emu()
    .verify_constraints()
    .build()
    .unwrap();
```

Runs the guest and checks the trace against all PIL constraints without generating a proof. Faster than full proving. Catches constraint violations that execution alone would miss.

## Setup

Call `.setup()` with the guest ELF to get the proving and verification keys:

```rust
use zisk_sdk::{include_elf, ElfBinary};

const GUEST_ELF: ElfBinary = include_elf!("my-guest-crate");

let prover = ProverClient::builder().emu().build().unwrap();
let (pk, vk) = prover.setup(&GUEST_ELF)?;
```

- **`pk`** (`ZiskProgramPK`): proving key, passed to `.execute()` and `.prove()`
- **`vk`** (`ZiskProgramVK`): verification key, passed to `.verify()`

ELF sources:

```rust
// From the include_elf! macro (compile-time embedding)
const GUEST_ELF: ElfBinary = include_elf!("my-guest-crate");

// From a file at runtime
use std::path::Path;
use zisk_sdk::ElfBinaryFromFile;
let elf = ElfBinaryFromFile::new(Path::new("/path/to/my-guest.elf"), false)?;
```

`include_elf!` is standard for host programs compiled alongside their guest. Use `ElfBinaryFromFile` for external or dynamically-loaded ELFs.

## Execute

Runs the guest and returns public outputs without generating a proof:

```rust
use anyhow::Result;
use zisk_sdk::{ElfBinary, ProverClient, ZiskIO, ZiskStdin, include_elf};

const GUEST_ELF: ElfBinary = include_elf!("my-guest-crate");

fn main() -> Result<()> {
    let prover = ProverClient::builder().emu().build()?;
    let (pk, _vk) = prover.setup(&GUEST_ELF)?;

    let stdin = ZiskStdin::new();
    stdin.write(&42u64);

    let result = prover.execute(&pk, stdin)?;
    let answer: u64 = result.get_public_values()?;
    println!("Guest returned: {}", answer);

    Ok(())
}
```

Fast. Skips proof generation entirely. The result also includes execution metadata like cycle count.

## Prove

Runs the guest and generates a proof:

```rust
let result = prover.prove(&pk, stdin).run()?;
prover.verify(result.get_proof(), result.get_publics(), &vk)?;
```

Default is VADCOP STARK. `.prove()` returns a `ProveBuilder`. Call `.run()` to execute. Chain proof-type selectors before `.run()`:

```rust
let result = prover.prove(&pk, stdin).compressed().run()?;
let result = prover.prove(&pk, stdin).plonk().run()?;
```

See [Proof Types](/proving-in-depth/proof-types) for compressed and PLONK details.

The `result` provides `result.get_proof()` and `result.get_publics()`. Always verify after proving. `.verify()` takes proof, public values, and verification key.

## Optional Builder Methods

Additional builder configuration methods.

### SNARK Support

```rust
let prover = ProverClient::builder()
    .asm()
    .snark()
    .build()
    .unwrap();
```

Enables SNARK circuit setup. Required before `.build()` if you intend to generate PLONK proofs. Skip it for VADCOP-only workflows to save initialization time.

::callout{icon="i-heroicons-exclamation-triangle"}
If you try to generate a PLONK proof without having called `.snark()` on the builder, proving will fail at runtime. This is the most common configuration mistake.
::

### Aggregation Mode

```rust
let prover = ProverClient::builder()
    .emu()
    .prove()
    .aggregation(true)
    .build()
    .unwrap();
```

Enables proof aggregation for combining multiple proofs into one.

### Verbose Logging

```rust
let prover = ProverClient::builder()
    .emu()
    .prove()
    .verbose(1)
    .build()
    .unwrap();
```

Verbosity level (`u8`): 0 = silent, 1 = standard, higher = more detail.

### Custom Proving Key Path

```rust
let prover = ProverClient::builder()
    .emu()
    .prove()
    .proving_key_path("/custom/path/provingKey")
    .build()
    .unwrap();
```

Override the default proving key artifact location.

### GPU Acceleration

```rust
let prover = ProverClient::builder()
    .emu()
    .prove()
    .gpu(None)
    .build()
    .unwrap();
```

Enables GPU-accelerated proving. `.gpu()` takes `Option<ParamsGPU>`. Pass `None` or `Some(ParamsGPU::default())` for defaults. Requires Linux with NVIDIA GPU and CUDA drivers.

Fine-grained control:

```rust
use zisk_sdk::ParamsGPU;

let prover = ProverClient::builder()
    .emu()
    .prove()
    .gpu(Some(ParamsGPU::default()))
    .build()
    .unwrap();
```

See [GPU Proving](/optimization/gpu-proving) for details.

## The Singleton Rule

`ProverClient` is a singleton: one per process. A second instance panics:

```rust
// This works
let prover1 = ProverClient::builder().emu().build().unwrap();

// This panics!
let prover2 = ProverClient::builder().emu().build().unwrap();
```

The backend manages global resources (memory-mapped files, GPU contexts) that can't be shared. To prove multiple programs, reuse one `ProverClient` with different `.setup()` calls.

## Complete Example

```rust
use anyhow::Result;
use zisk_sdk::{ElfBinary, ProverClient, ZiskIO, ZiskStdin, include_elf};

const GUEST_ELF: ElfBinary = include_elf!("fibonacci-guest");

fn main() -> Result<()> {
    // Build the prover with the ASM backend and SNARK support
    let prover = ProverClient::builder()
        .asm()
        .snark()
        .build()?;

    // Set up keys for this guest program
    let (pk, vk) = prover.setup(&GUEST_ELF)?;

    // Prepare inputs
    let stdin = ZiskStdin::new();
    stdin.write(&10u32); // compute the 10th Fibonacci number

    // Execute first to check correctness
    let exec_result = prover.execute(&pk, stdin)?;
    let fib: u32 = exec_result.get_public_values()?;
    println!("Fibonacci(10) = {}", fib);

    // Now generate a PLONK proof for on-chain verification
    let stdin = ZiskStdin::new();
    stdin.write(&10u32);
    let prove_result = prover.prove(&pk, stdin)
        .plonk()
        .run()?;

    // Verify locally before submitting on-chain
    prover.verify(prove_result.get_proof(), prove_result.get_publics(), &vk)?;
    println!("Proof verified successfully");

    // Save for later use
    prove_result.save_proof_with_publics("/tmp/fibonacci-proof")?;

    Ok(())
}
```
